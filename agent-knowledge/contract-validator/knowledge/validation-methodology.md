# 验收方法指南

> 本文是 contract-validator Prompt 的补充——VC 断言判定的详细规则和常见陷阱。

## 判定规则

### 逐条判定原则

每条 VC 断言独立判定，不允许：
- "整体不错，这条勉强算 PASS"
- "这条 FAIL 但影响不大，忽略"
- "这条和上条类似，一起判"

### PASS 条件

断言必须**客观上**满足：
- 检查的实际结果与断言的"通过标准"完全一致
- 不能"基本满足"、"大概符合"
- 数字比较用精确值（不是"差不多"）

### FAIL 条件

以下任一情况 = FAIL：
- 实际结果与断言矛盾
- 断言要求的行为未实现
- 断言要求的验证点无法观测（不可判定）

### UNCERTAIN 条件

以下情况标注 UNCERTAIN（不计入 PASS/FAIL，单独报告）：
- 断言描述模糊，无法客观判定
- 断言要求的验证环境不可用
- 断言依赖的外部状态不可控

## 歧义处理

| 断言描述 | 判定策略 |
|---------|---------|
| "返回正确的错误信息" | UNCERTAIN——"正确"是主观判断。标注建议：改为"返回含'无效参数'的错误响应" |
| "响应时间在合理范围内" | UNCERTAIN——"合理"无标准。标注建议：改为具体阈值 |
| "界面美观大方" | 不可判定——主观断言。标注建议：删除或用可测条件替代 |
| "用户操作流畅" | 同上 |

## 三类断言判定侧重

### 正常路径（VC-Nxx）
- 关注：功能行为是否正确
- 判定方式：给定合法输入，观察输出
- 常见陷阱：只测 happy path 一个输入

### 边界路径（VC-Bxx）
- 关注：极端输入下的行为
- 判定方式：空值/零值/最大值/超长值
- 常见陷阱：只测空值，忽略最大/超长

### 异常路径（VC-Exx）
- 关注：错误情况下的降级和恢复
- 判定方式：模拟错误条件
- 常见陷阱：只断言"不崩溃"，忽略"状态一致性"

## 报告规范

### PASS 报告
```
VC-N01: ✅ PASS
  检查: [简述检查了什么]
  证据: [实际观察到的结果，含具体值/状态码]
```

### FAIL 报告
```
VC-N01: ❌ FAIL
  断言: [原文]
  实际: [实际观察到的结果]
  差异: [断言要求 vs 实际的差异点]
```

### UNCERTAIN 报告
```
VC-N01: ⚠️ UNCERTAIN
  理由: [为什么无法判定]
  建议: [如何改进断言使其可判定]
```

## 常见陷阱

| 陷阱 | 表现 | 正确做法 |
|------|------|---------|
| 替断言解释 | "断言说'正确'就是返回非错误状态码" | 标注 UNCERTAIN，"正确"无客观标准 |
| 放松标准 | "虽然没到 200ms，但 210ms 也还行" | 严格按阈值判定 FAIL |
| 跳过不可测断言 | 直接标 PASS | 标注 UNCERTAIN + 理由 |
| 连带判定 | "N01-N05 都 PASS" | 逐条列出，每条带证据 |
| 推断未覆盖行为 | "从代码看应该没问题" | 只判定断言范围内的行为 |

## 验收结论格式

```
验收结论: PASS / FAIL

断言统计:
  正常路径(N): X/Y PASS
  边界路径(B): X/Y PASS
  异常路径(E): X/Y PASS
  不可判定: Z 条

如 FAIL: 列出所有失败断言ID + 差异概述
```
