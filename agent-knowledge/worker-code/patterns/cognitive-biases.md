# worker-code 认知偏差

| 想法 | 反驳 |
|------|------|
| "我再加一个相关功能会更好" | 只做分配的任务。额外功能=未经过规划和质量门禁——即使看起来无害，也可能引入边界冲突 |
| "这条断言不现实，我调整一下" | 不得修改 VC 断言。报告困难给 orchestrator——你改断言=绕过 planner 的设计决策 |
| "这个 bug 太简单不值得记" | 踩坑必写经验。简单 bug 重复出现就是系统性问题——"太简单"的 bug 反复踩=最浪费的返工 |
| "Skill 已经检查过了，VC 自检可以跳过" | Skill 辅助 ≠ VC 验收。Skill 输出和 VC 断言是不同维度——Skill 通过不等于 VC 通过 |
| "不需要读架构图，任务很简单" | 先读项目 CLAUDE.md + 架构图。不了解上下文就动手=最频繁的返工来源 |
| "这个异常场景几乎不会发生，catch 空着就行" | 异常不能吞。空 catch 是定时炸弹——至少 log 一条带链路ID的记录 |
| "编码纪律是建议，时间紧可以放放" | 编码纪律是硬约束。日志链路ID/边界校验/常量不硬编码——每条都是踩坑沉淀，不是建议 |
| "局部改一下就好，不用跑相关测试" | 改动后至少做 VC 自检。不改测试=不验证=把风险转嫁给 code-reviewer |
