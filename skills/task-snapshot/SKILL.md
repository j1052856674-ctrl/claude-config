# task-snapshot — 跨 Session 快启摘要生成器

## 输出格式（五段固定）

### 第一段：摘要句

一行精炼摘要，格式固定：

```
> **[项目名] [状态关键词] → [下一步]**
```

示例：
```
> **AI Worker Hub Iter4 全部完成**（F1+F2+F3+F4）→ 待功能实测
```

### 第二段：关键指针

表格形式，包含新 session 需要的所有定位信息：

```
> **关键指针**:
> | 维度 | 位置 |
> |---|---|
> | 项目路径 | `绝对路径/` |
> | 架构图 | `<project>/memory-hub/architecture-map.md` |
> | 当前计划 | `<project>/memory-hub/status/xxx-plan.md` |
> | 记忆目录 | `memory-hub/` |
> | 编译状态 | tsc X errors / cargo X errors / 非编译型项目 |
```

### 第三段：核心变更

精简的文件变更列表（只列文件名 + 一句话说明变更内容，不展开细节）：

```
> **核心变更**: file1(+说明) · file2(+说明) · file3(+说明)
```

### 第四段：待办事项

告诉新 session 接下来要做什么。格式为编号清单，每项一句话，标注优先级：

```
> **待办事项**:
> 1. [P0] xxx
> 2. [P1] xxx
> 3. [P2] xxx
```

优先级定义：
- **[P0]**：阻塞项，不做后续工作无法推进（如"修复编译错误"）
- **[P1]**：高优，下一步应该做的事（如"生成架构图"、"创建项目PRD"）
- **[P2]**：按需，有上下文时可做（如"补充监控告警配置"）

### 第五段：方法论收获

告诉新 session **学到了什么**——不是"做了什么"，而是"从中提炼出什么可复用的 insight"。格式为分类清单：

```
> **方法论收获**:
> - 设计原则: [可跨项目复用的架构/流程 insight]
> - 反模式: [我们纠正了什么错误模式 + 为什么它错]
> - 关键决策: [选了什么 + 为什么选它而不是替代方案]
> - 踩坑: [出错现象 → 根因 → 修正方式 → 下次怎么避免]
```

**各分类含义**：
- **设计原则**：可迁移到其他项目/场景的通用 insight（如"VC 文件是对话与编排的协议边界"）
- **反模式**：发现的系统性错误模式及其修正方向（如"Red Flags 与反驳表高度重复——两个板块功能重叠"）
- **关键决策**：不可逆或高成本的选择，附选择理由——以后有人质疑时有据可查
- **踩坑**：具体的出错→定位→修复→防范链。遵循 CLAUDE.md 踩坑沉淀规则

**分类判定规则**：如果一条 insight 同时描述正确做法和需纠正的错误，以正确做法为准归入"设计原则"，错误部分在描述中提及即可。同一条 insight 不重复出现在两个分类中。

**重要**：第五段是给人类和新 session AI 看的**摘要**——每个分类 1-3 条，每条一句话。完整内容必须写入对应知识文件（详见下方"自动记忆写入"）。

## 生成步骤

1. **读取项目上下文**：检查当前目录是否有 `memory-hub/`，读取 `MEMORY.md` 索引（若 MEMORY.md 不存在则跳过索引读取，后续 Step 7 写入时自动创建）。**若会话 system-reminder 已加载 MEMORY.md 且内容未过期，跳过重复读取——直接用会话上下文。**
2. **识别架构图**：在 `memory-hub/` 中找 `architecture-map` 或类似名称的文件。**非代码项目（文档工作区/Skill 设计库/PRD 工作区等）跳过此步**——代码项目才需要 architecture-map。
3. **识别计划文件**：找包含 `plan` 关键词的记忆文件
4. **确认编译状态**：按项目实际技术栈检测 — 有 `tsconfig.json` 则 `npx tsc --noEmit`，有 `Cargo.toml` 则 `cargo check`，有其他构建工具同理。多个构建工具并存时逐一运行，结果分项报告（如 `tsc: 0 errors, cargo: 2 errors`）。都不存在则写"非编译型项目"
5. **扫描 git 变更**（会话上下文优先，git 作交叉验证）：
   - 5a. **先盘点会话上下文**：列出本次对话中 Read/Write/Edit 操作涉及的所有文件。这些是本次 session 的**一手变更清单**，优先级高于 git 输出
   - 5b. 运行 `git rev-parse --git-dir 2>/dev/null` 判断是否 git 仓库。非 git 仓库则写"非 git 项目"，仅输出 5a 的会话上下文变更清单，跳过 5c/5d/5e
   - 5c. **降级判断**：运行 `git status --short | grep -v "^??" | wc -l` 统计已跟踪文件的变更数。若 **>20 个文件且大部分与会话上下文无关**（即 5a 清单中 <30% 出现在 git status 中），则 git 输出为历史噪音，降级为"基于会话上下文生成变更清单，git status 用于遗漏检测但不全量输出"
   - 5d. 正常模式（≤20 个或与会话高度相关）：运行 `git status --short | grep -v "^??" | grep -v "^warning:"` 输出已跟踪变更
   - 5e. 增量 diff：仅当满足以下所有条件时才执行 `git diff --stat HEAD~1`：①commit 数 ≥ 2 ②**代码项目**（非知识库/文档项目）③5c 未触发降级。否则写"跳过（非代码项目/降级模式）"
6. **组装五段输出**：按上述固定格式生成，确保所有路径是**相对于项目根目录**的（新 session cd 到项目路径后可直接读）。**第三段（核心变更）优先使用 5a 的会话上下文清单，5d 的 git status 仅作遗漏补充。**
7. **自动写入记忆**：组装第五段的同时，将每条方法论收获按路由表写入对应知识文件。**摘要文本输出和记忆文件写入是同一操作的两个产出，不得先输出摘要再异步补写文件。** 重大进展至少包含：①第五段列出的所有方法论条目 ②本次对话中做出的不可逆选择 ③新发现的架构理解。

## 自动记忆写入

生成摘要文本后，**必须**自动执行记忆写入操作，确保跨 session 知识不丢失。

### 写入路由

对本次对话中的重大进展，**按类型路由到不同位置**：

| 知识类型 | 写入位置 | bridge | 说明 |
|---------|---------|--------|------|
| 项目状态/阶段完成 | `<project>/memory-hub/status/` | true | 项目本地，即时可用 |
| 关键决策（选A不选B+理由） | `<project>/memory-hub/decisions/` | true | 决策上下文对后续工作至关重要 |
| 踩坑（出错→根因→修正→防范） | `<project>/memory-hub/lessons/` | true | 项目内重复踩坑时有据可查 |
| **设计原则/方法论**（跨项目可复用） | `<project>/memory-hub/decisions/`，标记 `bridge: true` | true | memory-bridge 后续沉淀到 `E:/个人仓库/03_Knowledge/记忆中枢/` |
| **反模式发现**（系统性错误+修正方向） | `<project>/memory-hub/lessons/`，标记 `bridge: true` | true | memory-bridge 后续沉淀到 `E:/个人仓库/03_Knowledge/记忆中枢/` |
| 评审框架/质量标准 | `<project>/memory-hub/reviews/` | 按需 | 项目本地评审资产 |
| 外部参考/工具配置 | `<project>/memory-hub/refs/` | 按需 | 工具链知识 |

**路由优先级**：
1. 项目特有的状态/决策/踩坑 → 项目本地 `memory-hub/`
2. 可跨项目复用的方法论/反模式/设计原则 → 先写 `<project>/memory-hub/` 并标记 `bridge: true`，再由 memory-bridge 沉淀到 `E:/个人仓库/03_Knowledge/记忆中枢/`
3. UAM 后续从项目 `memory-hub/` 路由 → dedup → 沉淀到 `E:/个人仓库/03_Knowledge/记忆中枢/`

**跨项目写入 fallback**：如果当前项目暂未建立 `memory-hub/`：
- 设计原则/反模式 → 回退写入项目本地 `<project>/memory-hub/decisions/`，标注 `bridge: true`
- memory-bridge 后续维护时统一沉淀到 `E:/个人仓库/03_Knowledge/记忆中枢/`
- 不要因为目标路径不存在而跳过写入

### 写入原则

- 只写非显而易见的（方案决策、踩坑、架构理解），不写代码结构和 git 历史
- 已有同类记忆则更新（将旧文件 `name` 写入新文件 `supersedes` 字段），不新建重复文件（遵循 CLAUDE.md supersedes 记忆合并规则）
- 指针而非全文（记忆文件存摘要+指针，单文件不超过 200 行，参照 CLAUDE.md 记忆文件上限规则）
- `bridge` 默认 `true`，仅在明确判断"无跨项目复用价值"时设 `bridge: false`（`bridge` 字段缺失时 memory-bridge 视为 `true`，与显式标注等效）
- **第五段方法论收获必须落盘**：摘要文本中的每一条方法论收获，都必须在对应位置有完整的知识文件。摘要只是索引，知识文件是正文。

## 重要约束

- **摘要不落盘，记忆落盘**：五段摘要文本仍是纯文本输出供复制粘贴；但记忆文件必须实际写入磁盘
- **第五段强制落盘**：方法论收获在组装摘要文本的同时写入知识文件（见 Step 7），不是先输出摘要再异步补写。摘要只是索引，知识文件是正文
- **方法论文档路由**：设计原则/反模式先写项目 `<project>/memory-hub/decisions/`，标记 `bridge: true`，再由 memory-bridge 沉淀到个人仓库（见路由优先级 fallback）；踩坑/决策→项目 `memory-hub/`
- **路径用相对路径**：关键指针中的路径用相对于项目根目录的路径（如 `<project>/memory-hub/architecture-map.md`），不用绝对路径
- **变更清单精简**：每个文件只写一句话说明，不超过 80 字。用 `·` 分隔，不换行
- **摘要句不超过 100 字**：只写项目名 + 状态 + 下一步，不加解释性文字
- **如果项目没有架构图**：指针中写"无（建议探索后创建）"，不要跳过这一行
- **如果项目无编译工具链**：编译状态写"非编译型项目"，跳过编译检查

## 何时使用

- 用户说"生成摘要"/"生成一句话"/"快启摘要"/"task snapshot"
- 阶段性任务完成，即将结束当前对话
- 用户希望新 session 能快速接手当前工作
- 对话即将因上下文过长被总结时，主动提醒用户可以生成摘要

## 何时不用

- 单次对话未产生实质性进展（仅问答、未修改文件、未做决策）
- 用户明确表示会继续在同一 session 中工作
- 项目刚初始化，尚无 `memory-hub/` 且无任何产出物可记录
