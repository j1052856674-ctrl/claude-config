# Phase 4: 代码与测试审查

## 目标

Phase 4 是 deep-review 的最终阶段，聚焦代码实现层面：逻辑正确性、安全边界、YAGNI 原则、性能、测试充分性、VC 断言覆盖和代码规范。本阶段同时吸收 `receiving-code-review` 的技术严谨哲学，将 push-back 方法论融入审查流程。

## 深度模式说明

| 模式 | 载入维度 | 适用场景 |
|------|----------|----------|
| 完整版 | 7 维度全开 | 新功能 / 重构 / 核心模块 |
| 简化版 | 3 维度（逻辑+安全+YAGNI） | 紧急修复 / 热补丁 / 小改动 |
| 最小版 | 仅逻辑+安全 | 仅改一行 / 配置变更 |

## 审查维度矩阵

| 维度 | 权重 | 载入条件 | 核心关注点 |
|------|------|----------|------------|
| 逻辑正确性 | 25% | 始终 | 边界条件、状态流转、并发安全、时序依赖 |
| 安全边界 | 20% | 始终 | 输入校验、权限检查、注入防护、敏感信息 |
| YAGNI 原则 | 10% | 非最小版 | 未使用代码、过度抽象、假设性未来需求 |
| 性能影响 | 15% | 非最小版 | 复杂度变化、N+1 查询、内存泄漏、热点路径 |
| 测试充分性 | 15% | 非最小版 | 关键路径覆盖、测试质量、回归保护、套件可运行 |
| VC 断言覆盖 | 5% | 非最小版 | 关键不变量、前置/后置条件、边界断言 |
| 代码规范 | 10% | 始终 | 命名一致性、注释充分性、可维护性、类型安全 |

---

## Phase 4 简化版定义

紧急修复场景下，仅执行以下 3 维度：

1. **逻辑正确性**：修复是否解决根因而非症状？边界条件是否处理？
2. **安全边界**：修复是否引入新攻击面？输入校验是否完整？
3. **YAGNI 原则**：修复范围是否最小化？是否附带无关改动？

---

## 测试充分性评分法

测试充分性评分由 4 个维度组成，总分 5 分：

| 维度 | 权重 | 说明 |
|------|------|------|
| 关键路径覆盖 | 40% | 核心业务流程、高频操作、关键分支是否被测试覆盖 |
| 测试质量 | 30% | 测试的断言是否充分、是否真正验证行为而非仅执行代码、Mock 是否合理 |
| 回归保护 | 20% | 本次改动相关的既有测试是否仍然通过、是否新增回归测试防止问题复发 |
| 套件可运行 | 10% | 测试套件是否能在本地顺利运行、是否存在 flaky 测试、CI 是否通过 |

### 评分判定

| 分值 | 等级 | 说明 |
|------|------|------|
| ≥ 4/5 | 🟢 通过 | 测试充分，可放心合并 |
| 3/5 | 🟡 警告 | 测试基本覆盖，但存在薄弱点，建议补充 |
| ≤ 2/5 | 🔴 不通过 | 测试严重不足，必须补充后才能合并 |

## VC 断言覆盖检查方法

VC（Verification Condition）断言用于在关键位置声明不变量，增强代码可信度。

### 检查清单

1. **关键不变量**：复杂数据结构在操作前后是否满足不变量条件？
2. **前置条件**：函数入口处是否对参数范围、类型、状态做前置断言？
3. **后置条件**：函数返回前是否对结果做后置断言？
4. **边界断言**：循环、递归、迭代器的边界条件是否被断言保护？

### 原则

- 断言是**防御性编程**，不是错误处理
- 断言在发布版本中可被移除，不应承担业务逻辑
- 每个关键路径至少有一处断言覆盖

## YAGNI 检查清单（3 条）

从 `receiving-code-review` 吸收的技术严谨哲学：

1. **未使用代码检查**
   - 代码中是否存在未被调用的函数、类、接口？
   - 是否引入了"未来可能用到"的抽象层？
   - 方法：在代码库中 grep 实际使用位置，无调用则移除

2. **过度实现检查**
   - 是否实现了当前需求以外的功能？
   - 是否引入了过度通用的配置/参数？
   - 是否假设了不存在的需求场景？

3. **假设性未来需求检查**
   - 代码注释中是否频繁出现"将来可能需要"、"预留扩展"等表述？
   - 抽象层是否基于假设而非实际需求？
   - 添加的复杂度是否对应已知的未来需求？

> **核心原则**："You Ain't Gonna Need It" — 未经验证的需求不值得代码复杂度。

## Push-Back 方法论（5 条核心原则）

从 `receiving-code-review` 吸收的技术反馈处理哲学，融入审查流程：

### 1. 验证先于接受

```
WHEN 收到审查反馈时：
  1. READ: 完整阅读，不立即反应
  2. UNDERSTAND: 用自己的话重述要求（或提问澄清）
  3. VERIFY: 对照代码库现实验证
  4. EVALUATE: 对此代码库是否技术上合理？
  5. RESPOND: 技术确认或有理有据的 push-back
  6. IMPLEMENT: 逐项实施，逐项测试
```

### 2. 禁止表演性认同

**绝不使用：**
- "You're absolutely right!"
- "Great point!" / "Excellent feedback!"
- "Let me implement that now"（验证前）

**替代方案：**
- 重述技术要求
- 提出澄清问题
- 用技术推理进行 push-back
- 直接开始工作（行动 > 言语）

### 3. 处理模糊反馈

```
IF 有任何项目不清楚：
  STOP — 不要实施任何内容
  ASK — 请求澄清

WHY: 项目之间可能存在关联。部分理解 = 错误实施。
```

### 4. Push-Back 的时机与方式

**何时 push-back：**
- 建议会破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对此技术栈技术上不正确
- 存在遗留/兼容性原因
- 与人类合作伙伴的架构决策冲突

**如何 push-back：**
- 使用技术推理，而非防御性态度
- 提出具体问题
- 引用工作中的测试/代码
- 涉及架构问题时请人类合作伙伴介入

### 5. 纠正性承认

如果 push-back 后发现是自己错误：

```
✅ "Checked [X] and it does [Y]. Implementing now."
✅ "Verified this and you're correct. My initial understanding was wrong because [reason]. Fixing."

❌ Long apology
❌ Defending why you pushed back
❌ Over-explaining
```

## 代码审查输出格式

审查结果使用 🔴🟡🟢 三色信号灯表格输出：

| 维度 | 状态 | 发现项数 | 摘要 |
|------|------|----------|------|
| 逻辑正确性 | 🟢/🟡/🔴 | N 项 | 一句话摘要 |
| 安全边界 | 🟢/🟡/🔴 | N 项 | 一句话摘要 |
| YAGNI 原则 | 🟢/🟡/🔴 | N 项 | 一句话摘要 |
| 性能影响 | 🟢/🟡/🔴 | N 项 | 一句话摘要 |
| 测试充分性 | 🟢/🟡/🔴 | N 项 | 一句话摘要 |
| VC 断言覆盖 | 🟢/🟡/🔴 | N 项 | 一句话摘要 |
| 代码规范 | 🟢/🟡/🔴 | N 项 | 一句话摘要 |

### 状态定义

| 符号 | 含义 | 处理建议 |
|------|------|----------|
| 🟢 | 无问题或仅建议性意见 | 可直接合并 |
| 🟡 | 存在需要关注的问题 | 建议修复后合并 |
| 🔴 | 存在阻塞性问题 | 必须修复后才能合并 |

## 测试充分性输出格式

测试充分性使用 4 维度评分表输出：

| 维度 | 得分 | 权重 | 说明 |
|------|------|------|------|
| 关键路径覆盖 | X/5 | 40% | 核心业务流程是否被覆盖 |
| 测试质量 | X/5 | 30% | 断言充分性、Mock 合理性 |
| 回归保护 | X/5 | 20% | 既有测试通过、新增回归测试 |
| 套件可运行 | X/5 | 10% | 本地/CI 可运行、无 flaky |
| **总分** | **X.X/5** | 100% | 🟢/🟡/🔴 |

### 判定规则

```
IF 总分 >= 4.0:
  状态 = 🟢 通过
ELIF 总分 >= 3.0:
  状态 = 🟡 警告
ELSE:
  状态 = 🔴 不通过
```
