# Phase 1: 需求澄清与价值剖析

**目标**：模糊需求 → 可验证的清晰需求 + 价值判断
**深度模式**：永不裁剪（Phase 1 始终深度评审）
**隔离模式**：Step 0 评分 ≥31 时触发全隔离——3 个 subagent 独立读需求，各产出"最关键的3个问题"，主对话找交集。详见 `references/multi-agent-isolation.md`。

---

## 评审维度矩阵

| 维度 | 检查什么 | 深度方法 | 评判标准 |
|---|---|---|---|
| **价值三轴** | 用户价值×可行性×风险 | 三轴独立评分(1-5)，最低轴决定总评 | 任一轴≤2 → 🟡质疑需求必要性 |
| **需求完整性** | 目标/场景/成功标准/边界 | 逐项检查是否明确可量化 | 缺成功标准 → 🔴；缺边界 → 🟡 |
| **假设风险** | 需求中隐含的假设 | 列出所有假设→标风险等级→标验证方式 | 高风险假设未标注 → 🔴 |
| **需求矛盾** | 内部自相矛盾 | 逐条对比需求描述 vs 成功标准 | 发现矛盾 → 🔴 |

---

## 价值三轴评分法

```
用户价值(1-5)：5=解决核心痛点 1=边缘功能没人用
可行性(1-5)：5=简单可靠 1=高复杂/高风险/技术未验证
风险(1-5)：5=无风险 1=安全/合规/数据风险显著

总评分 = min(用户价值, 可行性) × 风险
理由：最低轴决定天花板，风险越高总评越低（风险=乘法惩罚因子）

示例：
  用户价值=4, 可行性=3, 风险=5 → 3×5 = 15/25 ✅
  用户价值=4, 可行性=3, 风险=2 → 3×2 = 6/25  🟡
  用户价值=5, 可行性=2, 风险=5 → 2×5 = 10/25  🟡
  用户价值=5, 可行性=1, 风险=1 → 1×1 = 1/25   🔴
```

**总评分判定**：
- ≥12/25：✅ 需求有价值，进入 Phase 2
- 6-11/25：🟡 需求存疑，需补充论证
- ≤5/25：🔴 需求价值不足，建议重新评估

## 需求完整性检查清单

对每个需求逐项检查：

- [ ] **核心目标**：一句话描述，明确做什么
- [ ] **用户场景**：谁+什么情况+预期得到什么
- [ ] **成功标准**：怎么判断做对了？量化指标优先
- [ ] **约束条件**：技术/时间/资源限制
- [ ] **明确不做的**：边界清单，防止范围蔓延
- [ ] **关键假设**：每个假设有风险等级和验证方式

缺失项处理：
- 缺成功标准 → 🔴 严重（无法判断是否做对）
- 缺边界 → 🟡 中等（可能范围蔓延）
- 缺假设 → 🟡 中等（隐性风险未暴露）

|---|---|---|---|

高风险假设必须标注验证方式。未标注 → 🔴。

## 需求矛盾检查

逐条对比：
1. 需求描述 vs 成功标准 — 是否有"A要快又要精确"这类矛盾？
2. 不同需求条目之间 — 是否有"同时要求实时和批量"这类矛盾？
3. 需求 vs 约束条件 — 是否有"要求无限扩展但预算有限"这类矛盾？

发现矛盾 → 🔴 严重，必须指出并建议调和方案。

## 交付物验收完整性检查

**触发条件**：评审对象涉及可交付物（代码/API/文档产出/功能实现）。当需求描述中提到具体功能、接口、页面或数据产出时自动触发。纯探索/调研/分析类需求跳过。

### 四段式验收清单检查

| 段 | 检查内容 | 缺失影响 |
|----|---------|---------|
| **功能清单** | 所有用户故事/功能点是否以可验证方式列出？每个功能是否有明确的验收标准？ | 🔴 缺功能清单 → 无法确认范围完整性 |
| **接口/产出映射** | 每个功能是否映射到具体接口（API路径+方法）或文件产出（文件路径+格式）？ | 🟡 缺映射 → 有孤儿功能风险（功能描述存在但无实现载体） |
| **验证矩阵** | 每个接口/产出的 3 类路径（正常→边界→异常）是否都有验证用例？是否有可独立执行的验证方式（curl/Playwright/自动化脚本）？ | 🔴 缺验证矩阵 → 无法独立验收 |
| **非功能验证** | 安全/性能/兼容性/权限是否有验证项？ | 🟡 缺非功能 → 功能正确但可能上线即崩 |

### 检查流程

1. 扫描需求描述，识别所有功能点和接口/产出
2. 检查上述四段是否都有对应内容
3. 缺失项 → 在 Phase 1 输出中标注，建议用户补充
4. 如验收清单缺失严重（缺功能清单或缺验证矩阵）→ **给出 VC 文件骨架**：

```markdown
## VC 文件骨架（供 orchestrator 使用）

### 功能清单
| F# | 功能 | 用户故事 | Done标准 |
|----|------|---------|---------|

### 接口/产出映射
| API# | 方法 | 路径 | 承载功能 |

### 验证矩阵
| V# | 路径类型 | 输入 | 预期输出 | 验证方式 |
|----|---------|------|---------|---------|

### 非功能验证
| NF# | 维度 | 验证项 | 标准 |
|-----|------|--------|------|
```

**与 orchestrator 的交接**：VC 文件是 deep-review（对话阶段）与 orchestrator（编排执行阶段）的协议边界。deep-review 产出 VC 文件后，orchestrator 直接读取并启动 planner 细化断言，不再启动 requirements-reviewer。

## 敢反驳纪律

- 至少提出 **1 个用户未考虑的角度**
- 用户倾向某方案但评审认为另一个更好 → 明确写出否决理由
- 不迎合 = 不写"您选"，写"推荐 X，因为 Y"
- **反驳优先级**（按重要性排序）：
  1. 高风险假设反驳（假设错误→方向性偏差）
  2. 核心目标偏离反驳（方案偏离需求本质）
  3. 实现路径缺陷反驳（技术方案有可修正的坑）
  4. 命名/格式/细节反驳（最低优先级）
