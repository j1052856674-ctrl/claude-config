# Phase 2: 方案对比与决策推荐

**目标**：穷举方案 → 多维度评分 → 明确推荐
**深度模式**：永不裁剪（Phase 2 始终深度评审）

---

## 9维度评分矩阵

| 维度 | 默认权重 | 评分标准 |
|---|---|---|
| 用户体验 | 20% | 5=极致流畅 1=明显卡顿 |
| 技术可行性 | 18% | 5=简单可靠 1=高复杂/高风险 |
| 扩展性 | 12% | 5=可无感扩展 1=牵一发动全身 |
| 性能影响 | 8% | 5=无影响 1=明显劣化 |
| 学习成本 | 8% | 5=无需学习 1=需读文档 |
| 错误恢复 | 8% | 5=自动恢复 1=崩溃/丢数据 |
| 数据一致性 | 8% | 5=强一致 1=易冲突 |
| Token/上下文效率 | 10% | 5=极简高效（指令密度高、按需加载、报告写文件） 1=大量填充（教科书式展开、关联文件无节制拉取） |
| 隔离适配性 | 8% | 5=天然适合 subagent 隔离（独立验证任务、不需实时交互） 1=必须主对话交互（需实时决策、用户频繁介入） |

| Skill/文档评审 | Token效率 25%, 可行性 25%, 隔离适配性 15%, 学习成本 15%, UX 10%, 扩展性 10% | Skill 核心是执行效率和上下文经济性 |

## 敢反驳纪律

- 检查推荐方案是否 = 用户最初倾向 → 如是，必须有差异化理由
- 至少提出1个用户未考虑的角度
- 否决方案必须写一句话否决理由（不能只写评分低）
- 每个评分必须附具体检查操作（不能写"感觉还行"就给4分）

**反驳优先级**：
1. 高风险假设反驳
2. 核心目标偏离反驳
3. 实现路径缺陷反驳
4. 命名/格式/细节反驳

## 多视角隔离模式

**触发条件**：Step 0 隔离评分 ≥16 分（轻量隔离或全隔离模式）。

**核心原则**：同一个上下文内的"多视角"本质是角色扮演——共享证据基础、受确认偏误影响。真正的独立视角需要隔离上下文。详见 `references/multi-agent-isolation.md`。

### 执行流程

```
Step 2a: 按评审对象类型选择 2-3 个对抗视角
         （视角之间必须有天然冲突——如果大概率得出相同结论，隔离无意义）

Step 2b: 为每个视角写独立 subagent prompt
         （不同推理起点，不只换角色名——参见 multi-agent-isolation.md §三）

Step 2c: 并行启动 subagent → 每个输出独立评分矩阵 + 推荐方案

Step 2d: 主对话合成（四步法——参见 multi-agent-isolation.md §四）
         逐视角核心判断 → 交叉验证 → 置信度判定 → 合成报告
```

### 视角选择速查

| 评审对象 | 推荐视角 | 天然冲突 |
|---------|---------|---------|
| 架构决策 | 极简主义 / 未来维护者 / 替代路径 | 极简 vs 扩展性 |
| 安全设计 | 攻击者 / 合规审计 / 真实用户 | 安全 vs 体验 |
| 产品需求 | 用户价值 / 技术成本 / 业务影响 | 价值 vs 成本 |
| Skill/Agent | 使用者 / 维护者 / 编排者 | 简单 vs 完整 vs 标准 |
| 性能优化 | 基准测试 / 可读性代价 / 边际收益 | 性能 vs 可读性 |
| 配置/规范 | 新人视角 / 扩展视角 / 一致性视角 | 简单 vs 完整 vs 兼容 |

### 合成输出

合成报告必须包含：
- 推荐方案 + 置信度（🟢高/🟡中/🔴低）
- 各视角核心判断（≥1句/视角，标注共识/分歧）
- 分歧点说明（如适用）
- 集体盲区标注（如适用）

**置信度判定**：
- 3/3 一致 → 🟢 高置信度
- 2/3 一致 → 🟡 中置信度（必须列出少数意见）
- 各自不同 → 🔴 低置信度（强制用户介入）
- 2/2 一致 → 🟡 中置信度（样本小）
