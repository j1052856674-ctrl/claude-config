# 完整模式 PRD 完整性检查清单

> 生成完整模式PRD后，逐项对照此清单检查文档完整性。
> 每项标记 ✅（已满足）/ ⚠️（部分满足）/ ❌（缺失）。
> 必填项（带*）缺失则PRD不合格，必须补充后再提交开发。

---

## 第1章：需求背景与目标

| 检查项 | 必填 | 检查标准 |
|---|---|---|
| 1.1 需求背景 | * | 包含「现状」和「问题」两部分，2-5段文字描述 |
| 1.2 需求价值&目标 | * | 每条目标有量化指标或可判断的完成标准 |
| 1.3 衡量指标&推演逻辑 | * | 每个目标对应1-3个指标，有推演逻辑说明 |
| 1.4 相关方 | * | 列出所有涉及团队和人员（表格形式） |
| 1.5 上线时间及原因 | * | 有明确时间和背景原因 |
| 1.6 范围边界（Non-Goals） | * | 明确列出至少2项"不做"的内容 |

---

## 第2章：需求内容描述

| 检查项 | 必填 | 检查标准 |
|---|---|---|
| 2.1 目标 | * | 明确本期范围，如有分期规划写清各期关系 |
| 2.2 整体架构 | 否 | 有架构图（Mermaid）+ 模块职责表 |
| 2.3 整体流程 | 否 | 有流程图（Mermaid），覆盖正常+异常流程 |
| 2.4 触发场景 | 否 | 列出所有场景入口（表格形式） |
| 2.5 页面/功能详细规格 | * | **核心章节！** 有全局逻辑说明 + 逐元素交互规格表，每个元素有详细功能描述（不是笼统描述） |
| 2.6 AI能力需求（如适用） | 条件必填 | 涉及智能体时必须有话术设计表 |
| 2.7 安全与合规 | * | 必须有数据脱敏、权限隔离、操作记录三项内容 |

### §2.5 交互规格深度检查

| 检查项 | 检查标准 |
|---|---|
| 功能需求清单有功能ID编号 | 每个功能有FR-001等编号 |
| 验收标准有Given-When-Then格式 | 每个FR-ID至少1条验收条件 |
| 每个按钮都有 | 点击行为、禁用条件、加载状态 |
| 每个输入框都有 | 最大字数、快捷键、提示文案、校验规则 |
| 每个列表都有 | 展示规则、溢出截断、hover提示 |
| 每个上传功能都有 | 格式限制、数量限制、失败处理 |
| 有全局逻辑说明 | 页面权限、数据加载、跳转关系 |
| 功能描述不是笼统的 | "实现一个输入框"不合格，必须有逐字段详细规格 |

## 第4章：关键约束

| 检查项 | 必填 | 检查标准 |
|---|---|---|
| 关键约束表格 | * | 有5-8条业务级约束（性能、安全、合规等） |

## 第5章：数据需求概述

| 检查项 | 必填 | 检查标准 |
|---|---|---|
| 数据需求概述 | 否/可选 | 有数据表/实体概述（表名+核心字段+用途） |

---

## 第6章：需求变更记录

| 检查项 | 必填 | 检查标准 |
|---|---|---|
| 变更记录表 | 否 | 有空表格（后续变更时填写） |

---

## 总体质量评估

| 维度 | 评分标准 |
|---|---|
| 完整性 | 所有必填项都有实质内容 |
| 可执行性 | 产研可直接按此开发，不需要二次沟通确认 |
| 一致性 | 文字描述与架构图一致，功能间无逻辑冲突 |
| 行业专业性 | 风控项目有行业特有的安全/合规/审批内容 |

### 合格判定
- **全部必填项 ✅** → 合格，可进入开发
- **1-2个必填项 ⚠️** → 需补充后可进入开发
- **3+个必填项 ❌** → 不合格，必须补充完整
