# Frontend Skill v2 — 场景驱动 · 可视化工作台

> **核心理念**：AI 不懂"好品味"，只懂"从给定选项中选一个"。本 Skill 不教 AI 设计理论，而是给 AI 一组**具体的选择题**、**必须打勾的清单**、以及一个**用户在浏览器中看真实渲染的预览服务**。

## 触发条件

涉及前端页面生成就触发：建页面、做 landing page、写 dashboard、打磨 UI、改造已有网站、哪怕只说"做一个页面"。

不适用于纯后端任务。

---

## 执行流程总览

```
Step 1  场景判定（5场景）
Step 2  设计选配 + 🔌启动预览服务（会话级，只启动一次）
        → Creative 场景: 生成 style-picker.html → 用户选风格 → 默认色板+字体直出代码
        → 其他场景: 生成 picker.html → 用户选色板+字体
Step 3  场景门控组件清单
Step 4  代码生成（纪律规则）→ 写入 index.html
Step 4.5 ⛔ 硬门禁：自检报告 + impeccable（必须在对话中逐项输出，禁止跳过直接告知"完成"）
Step 4.6 🔧 自检通过后主动 offer 打磨选项（polish / bolder / quieter / distill）
Step 5  用户反馈 → 微调 → 覆盖写入 → 刷新即看 → 回到 Step 4.5
```

**每次代码改动后强制**：Step 4.5 自检（含 impeccable）必须在对话中完整输出，不只是内部检查。Step 4.6 自检通过后必须主动询问用户是否打磨，不等用户自己提。

**预览服务贯穿全程**：Step 2 启动后一直运行，用户随时刷新浏览器看最新结果。AI 通过对话接收用户反馈，修改后覆盖写入 `index.html`。

## Step 1: 场景判定 + 技术栈识别

根据用户描述判定场景，决定后续所有步骤的行为。

### 场景判定

| 场景 | 触发信号 | 策略关键词 | 后续行为 |
|------|---------|-----------|---------|
| **Dashboard/工具** | 后台管理、数据面板、配置页、自用工具、admin | 功能优先、信息密集、操作高效 | 不追问风格，功能色板+紧凑间距 |
| **Landing/品牌页** | 着陆页、官网、产品介绍、给客户看、营销 | 视觉冲击、品牌感、信任建立 | 可选品牌人格Q&A |
| **Creative/作品集** | 作品集、portfolio、创意、个人网站、设计感、与众不同 | 风格表达、视觉个性、记忆点 | **必问**风格方向 |
| **展示/PPT式** | 精美、演示、汇报、PPT风格 | 大字体、少文字、强层次 | 不追问，大字色板+展示字体 |
| **通用** | 无法判定 | 保守平衡 | 不追问，保守色板+务实字体 |

### 技术栈识别

检查项目已有文件：`tailwind.config.*` → Tailwind；`*.tsx`/`*.jsx` → React；`*.vue` → Vue；只有 `*.html` → 纯 HTML/CSS。

## Step 2: 设计选配 + 可视化预览

### 2.0 通用前置：Brief Inference（必做）

**在做任何色板/字体选择之前**，先从用户描述中提取信号，输出**设计判读 + 架构判读**：

```
🎯 设计判读：[页面类型] for [目标用户]，[风格关键词]语言，倾向 [设计方向]
🏗️ 架构判读：[单页/多页] · [纯静态/需后端/需CMS] · Section: [列表]
```

**A. 设计信号**（缺失的标注"未指定"，不猜测）：
1. **页面类型** — 先在下方速查表匹配品类，匹配不到再用通用分类（SaaS/消费品/代理/活动/作品集/编辑/重设计/...）
2. **风格词** — 用户说的形容词（极简/活力/专业/优雅/暗色/...）
3. **参考信号** — 用户提到的竞品/品牌/URL
4. **受众** — B2B采购/B2C消费者/招聘方/...
5. **约束** — 无障碍优先/公共部门/监管行业/儿童产品/...

**B. 产品架构信号**（成熟品类走默认值，非常规项目才追问）：
6. **架构形态** — 单页滚动 / 多页+导航
7. **数据能力** — 纯静态 / 需后端API / 需CMS后台
8. **内容结构** — 核心 section 列表及优先级

**架构推断纪律**：
- 先查下方品类速查表，命中则**直接套用默认架构，不追问**
- 品类模糊时，能合理推断的直接写在判读里，确认真歧义才问
- 不管命中还是推断，**最多只问一个问题**

**D. 内容确认（架构确认后必做，Step 4 代码生成前）**

架构确认后、进设计选配前，将 Section 清单展示给用户确认**每个 Section 放什么内容**：

```
📋 页面结构预览：

| Section | 内容 | 呈现形式 |
|---------|------|---------|
| Hero | 名字 + 一句话定位 | 居中大字 + 渐变 |
| 关于我 | 个人背景、方向、理念 | 左文右统计卡片 |
| 项目经历 | 公司/职业项目 | 时间线 |
| 个人实践 | 开源/ Side Project | 卡片 + 展开详情 |
| 技能 | 技术能力分类 | 标签云 |
| 联系 | GitHub / Email | 居中链接 |
```

**确认纪律**：
- 默认 Section 来自品类速查表，但**内容由用户决定**——用户可能想合并/拆分/重排
- 展示时附带"呈现形式"列，让用户对最终页面有预期（不只是文字列表）
- 用户调整后再次展示确认，直到用户说"可以"
- 用户说"按你说的来" → 用默认结构，但仍展示一次确认
- **此步骤不进入风格选配**——先确认结构，再选风格/色板/字体

**C. 常见品类默认架构速查表**（命中 → 套用默认值 → 展示判读 → 用户确认后进设计选配）：

| 品类 | 触发词 | 架构形态 | 数据能力 | 默认 Section 顺序 | 默认理由 |
|------|--------|:--:|:--:|------|------|
| 个人网站 | 个人网站、个人主页、个人介绍、about me | 单页 | 纯静态 | Hero → 关于我 → 项目/作品 → 经验/技能 → 联系 | 内容量 3-5 屏，单页滚动比多页跳转体验好；纯静态零费用零维护，改内容 push 即可 |
| Portfolio/作品集 | portfolio、作品集、showcase | 单页 | 纯静态 | Hero → 精选作品(grid) → 关于 → 联系 | 核心目的是快速展示作品，一页扫完比分类导航更高效；GitHub Pages/Netlify 免费托管 |
| SaaS Landing | SaaS、产品页、官网、landing | 单页 | 纯静态 | Hero → 功能 → 如何工作 → 定价 → 客户案例 → CTA → Footer | Landing 页本质是说服漏斗，单页叙事从认知→兴趣→行动一气呵成；静态托管+CDN 加载最快 |
| 技术博客 | 博客、blog、技术文章 | 多页 | 需CMS/SSG | 首页(文章列表) → 文章详情 → 关于 → 归档 | 每篇文章是独立长文，必须独立页面；内容持续增长需 CMS 管理；SSG 兼顾静态速度和动态内容 |
| 文档站 | 文档、docs、API文档、使用指南 | 多页 | 纯静态 | 首页 → 快速开始 → 指南 → API参考 | 文档按主题分页是业界标准（每页一个话题）；内容结构稳定，静态生成足够 |
| 个人Dashboard | 后台、管理面板、dashboard、自用工具 | 单页(SPA) | 需后端API | 侧边栏 → 数据概览 → 详情表 → 设置 | SPA 避免数据页切换时整页刷新；数据来自 API，必须有后端 |
| 活动/会议页 | 活动、大会、meetup、hackathon | 单页 | 纯静态 | Hero(日期+CTA) → 嘉宾 → 议程 → 赞助 → FAQ | 活动信息有限（1-2 屏），单页保证访客不迷路；静态足够，活动结束后不维护 |
| 在线简历 | 简历、resume、CV | 单页 | 纯静态 | Header → 经历 → 技能 → 项目 → 教育 | 简历就是一张纸的逻辑搬到网页，单页最自然；静态托管永久在线 |
| 产品更新日志 | changelog、更新日志、release notes | 单页 | 纯静态/需后端 | 时间线列表 → 订阅CTA | 本质是一个排序列表，单页时间线最直观；静态够用，要动态筛选才需后端 |
| 状态页 | status、服务状态、uptime | 单页 | 需后端API | 总体状态 → 服务列表 → 历史事件 | 状态数据必须实时查询 API；单页 dashboard 一目了然 |

**品类速查表使用规则**：
- 命中品类 → 输出 🏗️ 架构判读（含默认理由摘要），**展示给用户确认**："以上是根据「{品类}」品类推断的默认架构，这样可以吗？还是需要调整？"
- 用户确认 → 直接进设计选配
- 用户要求调整（如"多页"、"加后台"）→ 以用户说的为准，调整后再次展示确认
- 品类不在表中 → 从 3 个架构信号中挑需要问的，最多一个问题，拿到答案后同样展示确认

**反默认纪律**：不默认 AI-purple 渐变、不默认 Inter 字体、不默认三列等大卡片、不默认 cream/beige 背景。

根据 Step 1 判定的场景，走对应子流程：

| Dashboard | → 2.2 | 否 |
| Landing | → 2.3 | 可选（品牌人格） |

### 2.2 Dashboard 流程

**策略**：功能优先，不追问风格。

**色板候选**（选 3-4 个供 picker 展示）：
| 优先 | 色板名 | 理由 |
|:--:|--------|------|
| 1 | 蓝钢 `#2563EB` | SaaS/工具首选 |
| 2 | 墨绿 `#059669` | 金融/数据密集 |
| 3 | 暗黑 `#818CF8` | 长时间使用的暗色工具 |
| 4 | 石墨 `#374151` | 极简后台 |

**字体候选**（选 3-4 个）：
| 优先 | 字体 | 理由 |
|:--:|------|------|
| 1 | Inter + PingFang SC | 数据可读性最佳 |
| 2 | DM Sans + PingFang SC | 比 Inter 略有个性 |
| 3 | Space Grotesk + PingFang SC | 科技感工具 |
| 4 | JetBrains Mono（代码/数字） | 数据表格等宽数字 |

**密度预设**：紧凑（元素内边距 16px，卡片间距 12-16px）。

### 2.3 Landing 流程

**策略**：品牌表达优先，可选品牌人格 Q&A。

**品牌人格问询**（仅当用户描述中未明确风格方向时问）：
> "这个品牌更接近哪种感觉？极简克制(如 Linear) / 温暖手工(如 Patagonia) / 科技前卫(如 Vercel) / 经典专业(如 Stripe)"

**色板候选**（按人格映射）：
| 人格 | 推荐色板 | 主色 |
|------|---------|------|
| 极简克制 | 石墨 `#374151` 或 深靛 `#4F46E5` | 黑白灰+单强调色 |
| 温暖手工 | 暖土 `#D97706` | 暖橙/陶土 |
| 科技前卫 | 深靛 `#4F46E5` 或 海蓝 `#0891B2` | 饱和蓝紫 |
| 经典专业 | 海蓝 `#0891B2` 或 墨绿 `#059669` | 深蓝/墨绿 |
| 创意大胆 | 玫红 `#DB2777` | 玫红/粉 |

**字体候选**（按人格映射）：
| 人格 | 推荐标题字体 |
|------|------------|
| 极简克制 | Sora, Space Grotesk, Inter（此处允许） |
| 温暖手工 | Playfair Display, Instrument Serif, Satoshi |
| 科技前卫 | Space Grotesk, Syne, DM Sans |
| 经典专业 | DM Sans, Satoshi, Inter |
| 创意大胆 | Syne, Playfair Display, Instrument Sans |

**密度预设**：标准-宽松（元素内边距 24px，区块间距 48-64px）。

**Landing 专属规则**（Step 4 中强制执行）：
- Hero 必须适配首屏：标题 ≤2行/≤8词，副标题 ≤20词/≤4行，CTA 可见无需滚动
- Hero 顶部 padding ≤ `pt-24`
- Hero 内最多 4 个文字元素（Eyebrow/Brand strip 选一 + 标题 + 副标题 + CTA）
- Logo墙在 Hero **下方**，不在 Hero 内
- Logo墙只用 logo，不印行业/类别标签
- CTA 桌面端不换行（≤3词，理想 1-2词）
- 无重复意图 CTA（一个意图全页一个标签）
- 禁止三列等大功能卡片
- Section 布局不重复（同一布局最多出现 1 次）
- 每 3 个 Section 最多 1 个 Eyebrow
- 禁止 Zigzag 交替超过 2 个连续 Section
- 禁止 Split-Header（左大标题+右小段落）作为默认

### 2.4 Creative 流程

**策略**：风格表达优先，**视觉 picker 一步直出**（不串行两轮选择）。

#### 2.4.1 生成风格选择器

**不做文字问询**。直接生成 `style-picker.html`，展示 6 种风格的**视觉对比**（每种风格一个 200px 微型 Hero 预览 + 特征标签 + 代表品牌）：

| # | 风格 | 视觉特征 | 微型预览要素 | 代表 |
|---|------|---------|------------|------|
| 1 | **极简白** | 大量留白、细体标题、克制单色 | 左对齐、细字重标题、分割线、小号正文 | Linear, Apple |
| 2 | **粗野主义** | 高饱和底色、粗体大标题、raw 边框 | 纯黄/纯红底、900字重、硬边框元素 | brutalistwebsites.com |
| 3 | **编辑式** | 衬线体、网格线背景、非对称居中 | 衬线标题、垂直网格线、overline、装饰线 | NYT, Kinfolk |
| 4 | **暗色沉浸** | 深色底+微光晕、渐变文字、发光边框 | 暗底+径向光晕、渐变标题、发光 pill 标签 | Stripe, Vercel |
| 5 | **复古暖色** | 暖色底+纹理点阵、印章边框、装饰符 | 纹理背景、印章式标签、装饰符号分隔 | Patagonia, Filson |
| 6 | **实验性** | 非常规布局、霓虹色、变形元素、叠层 | skew 变形标签、叠层标题、模糊光斑背景 | Awwwards 实验类 |

**picker 交互**：点击卡片选中 → 底部 fixed 确认栏显示当前选择 → 点击"确认选择"。

#### 2.4.2 风格 → 默认色板+字体（一步直出，不串行）

用户选定风格后，**不生成第二轮 palette+font picker**。直接使用下表中的**默认组合**进入代码生成：

| 风格 | 默认色板 | 默认字体 | 备选色板 | 备选字体 |
|------|:--:|:--:|------|------|
| 极简白 | **石墨** `#374151` | **Sora** + PingFang SC | 蓝钢、深靛 | DM Sans, Inter |
| 粗野主义 | **暗黑** `#818CF8` | **Space Grotesk** + PingFang SC | 石墨、玫红 | Syne, JetBrains Mono |
| 编辑式 | **暖土** `#D97706` | **Playfair Display** + Noto Serif SC | 海蓝、石墨 | Instrument Serif, DM Sans |
| 暗色沉浸 | **暗黑** `#818CF8` | **Space Grotesk** + PingFang SC | 深靛、玫红 | Satoshi, Syne |
| 复古暖色 | **暖土** `#D97706` | **Satoshi** + PingFang SC | 玫红、墨绿 | Playfair Display, Instrument Sans |
| 实验性 | **玫红** `#DB2777` | **Syne** + PingFang SC | 暗黑、深靛 | Space Grotesk, Playfair Display |

**默认值选定原则**：优先选该风格最具辨识度的色板/字体（如暗色沉浸→暗黑底、编辑式→衬线体），而非通用安全牌。

#### 2.4.3 逃生舱：真实页面微调

代码生成后，在对话中输出：

> 🎨 用了「[默认色板名] + [默认字体名]」组合。不满意？告诉我换一套就行，比如"换深靛色板"或"试试 Syne 字体"——我会直接覆盖写入，你刷新即看。

**微调纪律**：
- 同风格下只换色板/字体（不重新生成整个页面），替换 CSS 变量 + Google Fonts 引用 + font-family
- 一次只换一个维度（要么色板要么字体），避免用户分不清哪个改动影响了什么
- 每次微调后提醒"刷新即看"
- 用户连续换 3 次仍不满意 → 建议重选风格（回到 2.4.1）

**密度预设**：宽松-极简（区块间距 64-96px）。

**Creative 专属规则**（Step 4 中强制执行）：
- 反中心对齐：VARIANCE 偏高时强制非对称布局
- 允许 Motion 强度 ≥6（允许 GSAP/Motion 复杂动效）
- 允许实验性字体配对
- 禁止默认三列等大卡片
- Section 布局不重复
- 每 3 个 Section 最多 1 个 Eyebrow

### 2.5 Presentation 流程

**策略**：叙事驱动，不追问。

**色板候选**：蓝钢、深靛、暗黑、石墨（高对比度优先）。

**字体候选**：Sora、DM Sans、Space Grotesk、Inter。

**密度预设**：极简（每屏一个核心信息，≤3句话）。

### 2.6 General 流程

**策略**：保守平衡，不追问。

**色板候选**：石墨、蓝钢、海蓝、墨绿。

**字体候选**：Inter、DM Sans、Satoshi、Sora。

**密度预设**：标准。

## Step 3: 组件清单——场景门控

### 3.1 通用必须（所有场景）

- [ ] 每个按钮有 `onClick` handler 或 `type="submit"`
- [ ] 每个 Modal/Dialog：打开方式 + 关闭方式（X按钮 + Esc + 点击遮罩）——**三种全有**
- [ ] 每个破坏性操作（删除/不可逆修改）：确认弹窗（AlertDialog），不依赖浏览器 `confirm()`
- [ ] 每个表单：`onSubmit` + loading 状态 + error 状态 + success 反馈
- [ ] 每个 Dropdown/Menu：打开 + 关闭（点击外部）+ 键盘导航（↑↓ Enter Esc）
- [ ] 每个 Tabs：点击切换 + 键盘导航（←→）
- [ ] 响应式四断点不掉落：375px / 768px / 1024px / 1440px
- [ ] 所有 CSS 变量/工具类已定义（无未定义引用）
- [ ] 图片 `alt` / icon按钮 `aria-label` / 输入框 `<label>` 关联
- [ ] 输入框不禁用粘贴（不设 `onPaste` 拦截）
- [ ] 空状态有明确下一步操作（非仅"暂无数据"）
- [ ] 可点击元素有 `cursor-pointer` + hover 反馈
- [ ] 触摸目标 ≥ 44px

### 3.2 Dashboard 场景附加

- [ ] 侧边导航（可折叠）或顶部导航 + 主内容区
- [ ] 表格/卡片列表：loading 骨架屏（匹配内容形状）+ empty 状态 + error 重试
- [ ] 搜索框：输入 → 触发搜索 → 展示结果（含空结果状态）
- [ ] 数据表格使用 `font-variant-numeric: tabular-nums`
- [ ] 高密度时去掉卡片容器，用分割线或间距区分（DENSITY > 7 时）

### 3.3 Landing 场景附加

- [ ] **Hero 适配首屏**：标题 ≤2行/≤8词，副标题 ≤20词/≤4行，CTA 可见无需滚动
- [ ] **Hero 顶部 padding ≤ pt-24**
- [ ] **Hero 内最多 4 个文字元素**（Eyebrow OR 品牌条 + 标题 + 副标题 + CTA，选≤4）
- [ ] **Hero 内禁止**：CTA下方小标签、信任条、"Used by"行、功能列表、社交证明头像行
- [ ] CTA 按钮：≤3词/标签，桌面端不换行
- [ ] 无重复意图 CTA（如"联系我们"和"取得联系"同时出现 → 只留一个）
- [ ] CTA 按钮链接到有效目标（不链向 `#` 或 `javascript:void(0)`）
- [ ] 至少一个社交证明区（客户 Logo / 数据指标 / 用户证言）
- [ ] Logo墙在 Hero **下方**独立区域，只用 logo（不印行业/类别标签）
- [ ] 页面有明确的视觉焦点引导（用户 3 秒内知道该看哪里）

### 3.4 Creative 场景附加

- [ ] 视觉个性明确（用户选的风格方向在整体布局中可辨识）
- [ ] 非对称布局（除非用户选极简白）
- [ ] 允许实验性排版（大字号跨度、非标准网格）
- [ ] 至少一处视觉记忆点（独特动效/大胆配色/非常规构图）

### 3.5 展示/PPT 式场景附加

- [ ] 大标题 + 极少正文（每屏 ≤ 3 句话）
- [ ] 强对比视觉层级（大小 × 颜色 × 留白）
- [ ] 每屏一个核心信息（不多信息并置）

### 3.6 常见 Section 交互模式速查

> **核心理念**：色板数据库管颜色、字体数据库管排版、本表管交互模式。AI 从给定选项中选，不自己发明交互方式。

| Section | 适用场景 | 推荐交互模式 | 避免 |
|---------|---------|------------|------|
| **联系区** | 个人网站 / Portfolio | 展示地址文字 + 图标复制按钮（不跳转、不 mailto:） | mailto: 链接、大号文字"复制"按钮、行间分割线 |
| **联系区** | SaaS Landing / 企业站 | 联系表单 或 mailto: + 社交媒体图标链接 | 纯文字地址无交互 |
| **Footer** | 个人网站 / Portfolio | 极简：一行版权 + 社交图标（无多列链接） | 多列站点地图、重复导航 |
| **Footer** | SaaS / 企业站 | 4-5 列链接组 + Newsletter 订阅 + 版权行 | 单行极简（信息密度不足） |
| **CTA 区** | Landing 页末 | 一个大主按钮 + 一个次按钮 + 一句信任标签 | 3+ 按钮并排、无信任标签 |
| **导航** | 单页滚动站 | 无顶部导航（锚点滚动即导航），或 sticky 极简条 | 多级下拉菜单、汉堡菜单 |
| **导航** | 多页站 | sticky 顶栏 + 核心链接 + CTA 按钮 | 超过 6 个导航项、无 CTA |
| **项目/作品卡片** | Portfolio / 作品集 | 统一行模板：左侧元信息 + 右侧内容，同类型 Section 共享 | 不同 Section 各用一种卡片风格 |
| **技能/标签区** | 个人网站 | 分组标签云（同类标签视觉统一） | 无分组的平铺列表、进度条/百分比 |
| **社交链接** | 任何场景 | 图标 only（无文字标签），hover 品牌色 | 图标 + 大字标签、垂直排列 |

**使用规则**：
- 代码生成前先查本表——涉及到的 Section 有对应条目时，必须采用推荐模式
- 无对应条目 → 选交互模式最接近的条目作为参考
- 表中"避免"列 = 硬禁止，与 Step 4 纪律规则同等效力

## Step 4: 代码生成——纪律规则

生成代码时遵守以下硬约束。规则编号供自检引用。

### 视觉纪律

| # | 规则 | 为什么 |
|---|------|--------|
| 1 | **禁止紫色/多色渐变** | AI 默认指纹 #1，专业前端极少用 |
| 2 | **禁止纯灰色**（`#808080`/`gray-*`） | 中性色必须 tint——偏暖或偏冷 |
| 3 | **禁止 `bounce`/`elastic` 缓动** | 过时业余感，用 `cubic-bezier(0.16, 1, 0.3, 1)` |
| 4 | **禁止 `glow`/`box-shadow` 扩散做交互提示** | 廉价感，用背景色变化或边框替代 |
| 5 | **禁止卡片嵌套卡片** | AI slop 视觉惰性——用间距、背景色差、分割线区分层级 |
| 6 | **禁止 emoji 当 icon** | 用 Heroicons / Lucide / Phosphor / Tabler |
| 7 | **颜色必须来自 Step 2 设计简报色板** | 禁止随意发明颜色 |
| 8 | **色彩一致性锁定** | 全页一个强调色，暖灰/冷灰不混用，一个圆角体系 |
| 9 | **禁止纯黑 `#000` / 纯白 `#fff`** | 用 off-black（`#111827` 等）/ off-white（`#F8FAFC` 等） |
| 10 | **标题 `text-wrap: balance`，正文 `text-wrap: pretty`** | 防止孤行 |

### 布局纪律

| # | 规则 | 为什么 |
|---|------|--------|
| 11 | **禁止 `h-screen`，用 `h-dvh` 或 `min-h-dvh`** | iOS Safari 地址栏折叠导致布局跳动 |
| 12 | **固定元素必须尊重 `safe-area-inset`** | 避免内容被刘海/底部指示条遮挡 |
| 13 | **禁止三列等大功能卡片** | AI slop 特征 #2，用 2列zigzag / 非对称网格 / 横向滚动替代 |
| 14 | **Section 布局不重复** | 同一布局类型全页最多 1 次。8 个 Section 至少 4 种不同布局 |
| 15 | **Zigzag 交替上限 2 个连续 Section** | 第 3 个连续 image+text split → 必须打破 |
| 16 | **禁止 Split-Header**（左大标题+右小段落） | 垂直堆叠标题+正文更清晰 |
| 17 | **Eyebrow 上限**：每 3 个 Section 最多 1 个 | 数 `uppercase tracking` 标签，超标即失败 |
| 18 | **Bento 格数 = 内容数** | N 个内容 → N 个格，无空格。至少 2-3 格有视觉变化（图片/渐变/图案） |
| 19 | **Logo墙在 Hero 下，只用 logo** | 不印行业标签，不放在 Hero 内 |
| 20 | **方形元素用 `size-*` 替代 `w-*`+`h-*`** | 更简洁，减少冗余 |
| 21 | **同类型相邻 Section 共享视觉模板** | 两个内容结构相似的相邻 Section（如项目精选+个人实践），用同一套卡片/行模板，仅通过密度（padding/间距/字号）区分层级，不换整体布局范式 |

### 动效纪律

| # | 规则 | 为什么 |
|---|------|--------|
| 22 | **动画只动 `transform`/`opacity`** | 不动 `width`/`height`/`top`/`left`/`margin`/`padding` |
| 23 | **必须尊重 `prefers-reduced-motion`** | 无障碍基本要求 |
| 24 | **交互反馈 ≤200ms** | 超过 200ms 觉得"卡" |
| 25 | **禁止在非动画元素上设 `will-change`** | 预分配 GPU 层，不用时浪费显存 |
| 26 | **循环动画离屏时必须暂停** | IntersectionObserver 或 CSS `animation-play-state` |
| 27 | **禁止 `useEffect` 做渲染逻辑能表达的事** | 用 `useMemo`/计算，`useEffect` 是副作用逃生舱 |

### 内容纪律

| # | 规则 | 为什么 |
|---|------|--------|
| 28 | **禁止伪造精确数字** | `92%`/`4.1×`/`48k` 除非来自真实数据 |
| 29 | **禁止滚动提示文字** | `Scroll down`/`↓ scroll`/`Scroll to explore` → 用户知道怎么滚动 |
| 30 | **禁止版本标签在 Hero 区** | `v0.6`/`BETA`/`EARLY ACCESS` → 除非是产品发布预告页 |
| 31 | **禁止 Section 编号** | `01 / About`/`002 · Process` → 用自然语言标题 |
| 32 | **禁止 em-dash (`—`)** | LLM 风格指纹。用句号/逗号/冒号替代 |
| 33 | **禁止装饰圆点** | 导航/列表/徽章前的彩色圆点 → 只用于真实语义状态 |
| 34 | **长列表禁止每行横线** | 10+ 行 `border-t`+`border-b` → 分组+稀疏分割线，或卡片替代 |
| 35 | **CTA 一个意图一个标签** | "联系我们"+"取得联系"+"开始项目" → 选一个 |

### 技术纪律

| # | 规则 | 为什么 |
|---|------|--------|
| 36 | **z-index 必须用固定尺度** | 定义语义层（dropdown→sticky→modal→toast），禁止 `z-50`/`z-999` |
| 37 | **使用已有主题 token / 组件优先** | 不重复造轮子，引入新颜色前先查项目已有 token |
| 38 | **禁止 `window.addEventListener('scroll')`** | 用 Motion `useScroll()`/ScrollTrigger/IntersectionObserver/CSS scroll-driven |
| 39 | **`cn()` 工具合并 class**（`clsx`+`tailwind-merge`） | 避免样式冲突 |
| 40 | **中文页面必须有中文字体栈** | `"PingFang SC", "Microsoft YaHei", "Noto Sans SC", sans-serif` |
| 41 | **Hero 必须有真实视觉元素** | 禁止纯文字+渐变背景做 Hero——用真实图片/生成图片/组件预览 |

### 5.3 设计微调（自检通过后主动 offer，不等用户问）

> **纪律**：自检全部 ✅ 后，AI 必须主动在对话中列出可选的打磨方向，让用户选择。禁止跳过此步骤直接说"完成"。

| 用户说 | 操作 | 具体做法 |
|--------|------|---------|
| "打磨一下"/"润色" | **polish** | 检查间距节奏、对齐精度、hover 过渡、阴影 tint、排版层级，逐项收紧 |
| "太素了"/"单调" | **bolder** | 加大字号跨度、增加强调色面积、强化动效节奏、加一个视觉记忆点 |
| "太花哨了"/"太吵" | **quieter** | 减动画、收强调色、统一下沉色调、去装饰元素 |
| "简化一点"/"太多东西" | **distill** | 去除非必要元素、合并重复信息、减少视觉层级到 ≤3 层 |

**主动 offer 模板**（自检通过后必须输出）：

> ✅ 自检通过。是否需要打磨？可选方向：**polish**（收紧间距/过渡）· **bolder**（强化视觉冲击）· **quieter**（收敛更安静）· **distill**（精简约化）——或者直接确认完成。 |

微调后覆盖写入 `index.html`，用户刷新即看。

### 5.3.1 迭代纪律（反馈驱动修改时的强制约束）

用户反馈驱动的修改中，最大风险是**修一个元素时重写整个页面，导致已确认的设计静默丢失**。

| # | 规则 | 为什么 |
|---|------|--------|
| I1 | **只改用户明确要求的元素** | 用户说"改字体"→ 只换字体引用 + font-family，不碰布局/色板/结构 |
| I2 | **局部编辑优先，禁止全文重写** | 用 Edit 工具做精准替换，非用户要求对整体设计不满意时禁止 Write 全量覆盖 |
| I3 | **改前确认改动范围** | "改了 X，不会动 Y 和 Z"——让用户知道哪些东西保持原样 |
| I4 | **保留已确认的设计锚点** | 上一轮用户认可的视觉元素（统计卡片、CTA 按钮、Section 布局）视为已锁定，迭代中不主动改动 |
| I5 | **违反内容纪律的默认值立即修正** | 当前 SKILL.md 内容纪律（§28-32）已明确禁止滚动提示、Section 编号、em-dash、装饰圆点——生成时违反属于自检失败 |
| I6 | **涉及多 Section 重设计时先生成独立预览** | 生成 `section-preview.html` 只包含待改 Section，隔离迭代、反复刷新确认；通过后再合入 index.html。避免全局重写导致已确认区域视觉退化 |

**迭代反模式**（发生过，记录以防范）：

| 反模式 | 案例 | 正确做法 |
|--------|------|---------|
| 修小改大 | 用户要"去 nav + 调内容结构"，AI 连 Hero CTA 按钮 + 内容宽度 + 动效全改了 | 明确改动边界：只改 nav + Section 顺序，其余保持原样 |
| 全文重写 | 每次微调用 Write 全量覆盖 index.html | 用 Edit 做精准替换；只有用户对整体不满意时才全文重写 |
| 静默退化 | 重写时丢失了统计卡片、渐变色标题、CTA 按钮等已确认的视觉锚点 | 重写前盘点上一版的视觉锚点清单，写入新版本时逐项核对 |
| 全页迭代局部问题 | 用户对 2 个 Section 不满意，AI 重写整页 → Hero/About/Contact 被意外改动 | 生成 section-preview.html 隔离迭代 → 用户确认 → Edit 精准替换目标 Section |

## 反模式速查（生成时必避）

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| 紫色渐变 | `from-purple-500 to-pink-500` | 从色板选主色做单色渐变或实色 |
| 纯灰文字 | `text-gray-500` / `#808080` | 用色板的文字辅色（已 tint） |
| bounce 动画 | `animation: bounce 2s infinite` | `transition: 200ms cubic-bezier(0.16, 1, 0.3, 1)` |
| glow hover | `box-shadow: 0 0 20px rgba(...)` | `background-color` 微调或 `border-color` 变化 |
| 卡片嵌套 | `<Card><Card><Card>` | 最外层卡片 → 内部用 `bg` 变化 + `border` + 间距 |
| Inter 标题 | Landing/Creative 页标题用 Inter | 从字体表选有性格的标题字体 |
| 三列等大卡片 | icon+heading+text × 3 水平排列 | 2列zigzag / 非对称网格 / 横向滚动 |
| 空状态死胡同 | 空列表显示"暂无数据"无操作 | 加"创建第一个"或"去探索"按钮 |
| 无 hover 状态 | 可点击元素无 hover 样式 | `cursor:pointer` + bg微调 + `transition` |
| h-screen 移动端跳动 | `h-screen` / `100vh` | `h-dvh` / `min-h-dvh` |
| 交互反馈过慢 | transition >300ms 或 ease-in | 150-200ms + ease-out |
| useEffect 做渲染 | `useEffect(() => setX(calcY), [y])` | `const x = useMemo(() => calcY, [y])` |
| 离屏动画白耗 GPU | 无限循环动画在视口外持续运行 | IntersectionObserver + `animation-play-state: paused` |
| 长列表横线堆砌 | 10+ 行每行 `border-t`+`border-b` | 分组 + 稀疏分割线，或卡片替代 |
| CTA 重复意图 | "联系我们"+"取得联系"+"开始项目"同时出现 | 一个意图只留一个标签，全页统一 |
| Section 布局重复 | 3个Section用同一种 layout | 至少 4 种不同布局类型覆盖 8 个 Section |
| Eyebrow 泛滥 | 每个 Section 都有 uppercase tracking 标签 | 每 3 个 Section 最多 1 个 |
| 滚动提示 | `Scroll down` / `↓ scroll` / 鼠标动画 | 用户知道怎么滚动，去掉 |
| 版本标签 | Hero 中有 `v0.6`/`BETA` 标签 | 除非是产品发布预告页 |
| em-dash | 标题/正文中 `—` 作为分隔/停顿 | 句号/逗号/冒号替代 |
| Split-Header | 左大标题 + 右小段落作为 Section 头 | 垂直堆叠标题+正文 |
| Hero 纯文字 | 只有文字+渐变背景，无真实视觉元素 | 用真实图片/生成图片/组件预览 |
| 假精确数字 | `92%`/`4.1×` 无数据来源 | 有真实数据才写数字，否则用定性描述 |

## 预览服务与文件管理

### 目录约定

所有预览文件写入项目根目录的 `.claude/preview/`：

```
<项目根目录>/.claude/preview/
├── style-picker.html  # Creative 场景风格选择器（Step 2.4 生成）
├── picker.html        # 其他场景色板+字体选择器（Step 2.11 生成）
└── index.html         # 生成的页面（Step 4 生成，每次覆盖）
```

### 文件纪律

- **固定文件名**：`style-picker.html`（Creative 场景）、`picker.html`（其他场景）和 `index.html`，不版本后缀
- **每次覆盖**：不产生 `index-v2.html`、`index-final.html`、`index-v3-fixed.html` 等
- **`.gitignore`**：项目应添加 `.claude/preview/` 到 `.gitignore`

### 预览服务

```bash
# 启动（Step 2 执行一次，会话级）
node .claude/skills/frontend/scripts/dev-server.mjs
```

- 零外部依赖，仅 Node.js 内置模块
- 自动端口检测（默认 3333，被占用则递增）
- 后台运行，不阻塞 AI 继续工作
- 服务已在运行时不重启，直接覆盖文件即可

### 迭代工作流

```
AI 生成/修改代码 → 覆盖写入 index.html → 告知用户 → 用户刷新浏览器
用户反馈"按钮颜色不对" → AI 修改 → 覆盖写入 → 用户刷新
用户说"打磨一下" → Step 5.3 polish → 覆盖写入 → 用户刷新
```
