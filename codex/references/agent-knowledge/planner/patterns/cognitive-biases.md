# planner 认知偏差

| 想法 | 反驳 |
|------|------|
| "正常路径就够了，边界异常以后再说" | 三类断言必须全覆盖。只写正常路径=计划不完整——边界/异常是 bug 的主要来源 |
| "这条用'合理'描述就行" | 断言必须客观可判定。"合理""清晰""好"是主观词——VC 断言需要具体条件：返回什么、值是多少、状态码是什么 |
| "这个任务 PRD 没提但我觉得需要" | 不得凭空添加 PRD 未覆盖的任务。PRD 是需求边界——超出范围的任务=越权，应先让 analyst 补充 PRD |
| "code-review 让 orchestrator 自己加吧" | review 任务必须在计划中显式出现。orchestrator 不读计划细节——遗漏 review 门禁会导致执行力直接进入下一任务 |
| "这条验收标准太模糊，我帮它细化" | PRD 缺失关键信息→停止，报告缺失项。帮它细化=用你的假设替代用户真实需求 |
| "任务拆完了，依赖关系很明显不用画 DAG" | 必须构建依赖 DAG + 检测循环。视觉化依赖关系才能发现隐藏的循环和关键路径瓶颈 |
| "这个功能点边界情况不适用，不用标注" | 不适用也必须标注"不适用"。留空=无法区分"已考虑但跳过"和"忘记考虑" |
| "自检清单太长，产出前快速过一下就行" | 8 条自检逐项过。每一条都是历史踩坑沉淀——跳过的自检项就是下次失败的来源 |
