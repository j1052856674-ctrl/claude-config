二、流程图强制规范要求

✓ 强制要求 — 所有 PRD 流程图必须遵守
  1. 只使用 Mermaid 标准图表类型：flowchart、sequenceDiagram、stateDiagram-v2，不使用自定义图形
  2. 流程图必须覆盖完整链路：从触发起点到业务终点，不能只有一半流程
  3. 判断节点必须标注所有分支：菱形节点每个出口必须有标签（通过/失败/是/否/需补充等）
  4. 异常流程不能遗漏：失败、超时、降级、回退等异常分支必须画出
  5. 流程图与文字描述必须一致：图中的步骤与交互规格的描述不能矛盾
  6. 节点命名使用业务语言：用"用户发起对话""策略判断"等业务术语，不用"Step1""NodeA"等技术编号
  7. 连线标签必须写明流转条件：不能只画箭头不写为什么流转
✗ 禁止 — 以下模式不允许出现在 PRD 流程图中
  1. 禁止使用 personal_knowledge_base 中的自定义图形模式 — 如四阶段架构图（带自定义配色方案的认知层/执行层/推理层/交付层）
  2. 禁止使用自创视觉模式 — 如自定义颜色分层、自创图层命名、非标准节点形状
  3. 禁止使用 memory 存储的图形偏好 — 流程图样式不依赖个人记忆/偏好
  4. 禁止只画主干流程忽略异常分支 — 只画"正常通过"路径是不合格的
  5. 禁止节点命名使用抽象编号 — "Step1-Step2-Step3"是不合格的，必须用业务语言
  6. 禁止使用非 Mermaid 图表语法 — 不用 PlantUML、Graphviz 等其他工具语法

三、允许使用的 Mermaid 图表类型

图表类型Mermaid 语法适用场景优先级
业务流程图 flowchart TD 或 flowchart LR 展示业务流程从起点到终点的完整链路，含判断分支和异常处理 首选
跨职能流程图（泳道图） flowchart TD + subgraph 多个角色/系统参与的流程，需要标注各方职责边界 首选
时序图 sequenceDiagram 展示系统间交互顺序（前端-后端-数据库-三方等） 补充
状态图 stateDiagram-v2 展示对象/业务的状态变迁（如订单状态：待支付-已支付-已完成） 可选

四、标准节点形状规范

节点类型Mermaid 语法用途示例
开始/结束 A(([开始])) 标记流程起点和终点 
A(([用户进线])) --> B[意愿确认]
流程步骤 B[步骤名] 表示一个具体的业务操作步骤 
B[意愿确认] --> C{客户意愿}
判断/分支 C{判断条件} 表示需要判断的节点，必须标注所有出口条件 
C -->|愿意| D[材料收集]
C -->|不愿意| E[礼貌结束]
子流程 B[[子流程名]] 表示可展开的子流程（引用其他详细流程图） 
B[[规则挖掘详细流程]] --> C{规则是否有效}

连线规范

连线类型Mermaid 语法用途要求
带标签连线 A -->|标签| B 标注流转条件 所有判断节点的出口必须用带标签连线
普通连线 A --> B 顺序流转（无需条件判断） 仅用于无条件的顺序步骤之间

五、标准流程图示例

示例1：风控 Agent 业务流程（泳道图）

flowchart TD
    subgraph 用户侧
        A(([策略人员进线])) --> B[输入策略需求]
    end

    subgraph Agent侧
        B --> C[大模型理解意图]
        C --> D{需求是否明确}
        D -->|明确| E[挖掘策略规则]
        D -->|不明确| F[追问澄清]
        F --> B
        E --> G{规则是否有效}
        G -->|有效| H[输出规则+唯一标记]
        G -->|无效| I[调整挖掘方向]
        I --> E
    end

    subgraph 系统侧
        H --> J[规则配置到米霍克]
        J --> K[规则上线+效果追踪]
    end

    K --> L(([流程结束]))

示例2：回捞 Agent 对话流程

flowchart TD
    A(([客户进线])) --> B[意愿确认]
    B --> C{客户意愿}
    C -->|愿意配合| D[材料收集]
    C -->|不愿意| E[礼貌结束]
    C -->|犹豫| F[进一步引导]
    F --> B
    D --> G{材料是否完整}
    G -->|完整| H[额度评估]
    G -->|不完整| I[追问补充材料]
    I --> D
    H --> J{评估结果}
    J -->|通过| K[告知额度+推送卡片]
    J -->|拒绝| L[礼貌婉拒+结束]
    J -->|需补充| M[推荐新材料]
    M --> D
    E --> N(([结束]))
    L --> N
    K --> N

示例3：系统交互时序图

sequenceDiagram
    participant U as 用户
    participant F as 前端
    participant A as Agent平台
    participant M as 大模型
    participant D as 决策引擎

    U->>F: 输入策略需求
    F->>A: 发送对话请求
    A->>M: 调用模型推理
    M-->>A: 返回规则建议
    A->>A: 规则校验+打唯一标记
    A-->>F: 返回规则+配置按钮
    F-->>U: 展示规则结果
    U->>F: 点击规则配置
    F->>D: 规则部署请求
    D-->>F: 部署成功
    F-->>U: 配置完成提示

六、禁止使用的模式

✓ 标准模式（允许）

  • flowchart TD/LR — 标准业务流程图
  • sequenceDiagram — 系统交互时序图
  • stateDiagram-v2 — 状态变迁图
  • 标准节点形状：矩形、菱形、圆形
  • 连线标签标注流转条件
  • 业务语言命名节点

✗ 自创模式（禁止）

  • 自定义配色方案的分层架构图（如认知层紫色/执行层蓝色等）
  • personal_knowledge_base 中存储的自创图形模式
  • memory 中记录的图形偏好
  • 非标准节点形状
  • PlantUML / Graphviz 等其他语法
  • 抽象编号命名（Step1/NodeA）

七、流程图质量自检清单

#检查项标准判定
1图表类型使用了 flowchart / sequenceDiagram / stateDiagram-v2是/否
2完整链路从起点到终点覆盖完整流程，无遗漏环节是/否
3判断分支标注所有菱形节点的出口都有标签是/否
4异常流程覆盖至少包含1个异常/失败分支是/否
5节点命名使用业务语言，非抽象编号是/否
6图表与文字一致图中的步骤与交互规格描述对应是/否
7无自定义图形未使用 knowledge_base/memory 中的自创模式是/否
8Mermaid语法正确语法可正确渲染，无错误是/否
