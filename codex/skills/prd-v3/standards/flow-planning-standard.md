# 流程规划阶段自审规范 v1.0

> 定位：流程规划文档生成后的必经审查环节，确保流程逻辑正确性，避免缺陷传递到后续阶段。
> 前置条件：流程规划文档（节点清单 + 连线清单 + 异常分支定义）已输出

---

## 一、审查层级定位

```
流程规划文档生成
      ↓
┌─────────────────────────────┐
│ 🔍 第一层：流程规划逻辑自审   │ ← 本文件定义
│   • 检查项：流程闭环、异常完整、追问机制、节点粒度
│   • 核心问题："流程对吗？"
└─────────────────────────────┘
      ↓（通过后）
蓝图生成
      ↓
┌─────────────────────────────┐
│ 🔍 第二层：蓝图描述自审       │ ← diagram-drawio.md §C7.5
│   • 检查项：节点定义完整、连线覆盖、与PRD一致性
│   • 核心问题："描述全了吗？"
└─────────────────────────────┘
      ↓（通过后）
drawio 生成 → PNG 导出
      ↓
┌─────────────────────────────┐
│ 🔍 第三层：DrawIO 图形自审    │ ← diagram-drawio.md §C7.5
│   • 检查项：连线交叉、节点对齐、配色、泳道间隙
│   • 核心问题："图画得对吗？"
└─────────────────────────────┘
```

**关键原则**：流程规划阶段的逻辑错误修复成本最低（改几个字），如果流到后面才发现，修复成本成倍增加。

## 二、自审触发时机

```
流程规划文档（节点清单 + 连线清单 + 异常分支）
        ↓
进入流程规划逻辑自审（本阶段）
        ↓
    ┌───────────┐
    │ 通过      │ → 锁定流程规划 → 进入蓝图阶段
    │ 不通过    │ → 修正流程规划 → 重新自审（最多3次）
    └───────────┘
```

**最大迭代上限**：3 次。超过暂停并说明。

## 三、自审检查清单

### 3.1 流程完整性（基础项）

| # | 检查项 | 标准 | 判定 |
|---|-------|------|------|
| 1 | **起点明确** | 有且仅有 1 个开始节点（椭圆） | ✅/❌ |
| 2 | **终点明确** | 有至少 1 个结束节点（椭圆），且所有路径最终都能到达结束 | ✅/❌ |
| 3 | **路径覆盖** | 从起点到终点的每条路径都已定义，无悬空分支 | ✅/❌ |
| 4 | **节点连通** | 所有节点都有入线和出线（起止节点除外），无孤立节点 | ✅/❌ |
| 5 | **命名规范** | 节点名称使用"动词+名词"格式（如"选择角色"），无"XX完成"等后缀 | ✅/❌ |

### 3.2 异常分支完整性（核心项）

| # | 检查项 | 标准 | 判定 |
|---|-------|------|------|
| 6 | **异常覆盖** | 所有可能的异常场景（超时/失败/降级/参数非法/网络异常）都有分支处理 | ✅/❌ |
| 7 | **异常闭环** | 每个异常分支都有明确的**落地节点**（不悬空），最终能到达结束或回到主流程 | ✅/❌ |
| 8 | **异常兜底** | 每个异常分支都有明确的**兜底处理方案**（重试/降级/提示/人工介入） | ✅/❌ |
| 9 | **异常重试** | 超时/失败类异常有明确重试策略（自动重试几次？手动重试入口？） | ✅/❌ |
| 10 | **异常收敛** | 避免异常分支无限扩散，尽量归拢至统一处理节点 | ✅/❌ |
| 11 | **异常编码** | 异常分支建议标注唯一编码（如 ERR-001），便于追踪和文档对照 | ✅/❌ |
| 12 | **不可静默失败** | 兜底处理必须被监控和记录，禁止"静默吞掉"异常 | ✅/❌ |

### 3.3 追问机制完整性（核心项）

| # | 检查项 | 标准 | 判定 |
|---|-------|------|------|
| 13 | **追问触发** | 追问的触发条件和入口位置已明确 | ✅/❌ |
| 14 | **追问退出** | 追问有明确的退出条件和退出路径（点击关闭/超时/完成） | ✅/❌ |
| 15 | **追问轮次** | 追问有明确的轮次上限（如 50 轮）和超限处理方案 | ✅/❌ |
| 16 | **追问超时** | 追问有明确的超时阈值（如 30 秒）和超时处理方案 | ✅/❌ |
| 17 | **追问循环** | 追问循环路径清晰（回到追问输入/回到报告页/结束） | ✅/❌ |
| 18 | **追问历史** | 追问历史记录的保留策略、清理策略、脱敏策略已明确 | ✅/❌ |

### 3.4 边界条件（核心项）

| # | 检查项 | 标准 | 判定 |
|---|-------|------|------|
| 19 | **输入校验** | 所有用户输入点（选择角色、追问输入）都有前置校验 | ✅/❌ |
| 20 | **前置条件** | 每个操作的前置条件已明确（如"查询报告"前必须"已选角色"） | ✅/❌ |
| 21 | **后置条件** | 每个操作的后置条件已明确（如"报告生成"后进入"展示报告"） | ✅/❌ |
| 22 | **并行处理** | 并行操作的触发条件和汇聚条件已明确 | ✅/❌ |
| 23 | **网络异常** | 前端网络断开/弱网的检测和处理方案已明确 | ✅/❌ |
| 24 | **服务降级** | 下游服务不可用时的降级策略（返回默认值/缓存/友好提示）已明确 | ✅/❌ |

### 3.5 节点粒度（一般项）

| # | 检查项 | 标准 | 判定 |
|---|-------|------|------|
| 25 | **同类合并** | 同类操作已合并（如"数据检索"合并了"查询字段含义+查询表结构"） | ✅/❌ |
| 26 | **不同类拆分** | 不同类操作已拆分（如"角色校验"和"确定查询范围"不能合并） | ✅/❌ |
| 27 | **Agent 展开** | Agent 内部流程是否需要在总流程图中展开？（通常需要） | ✅/❌ |
| 28 | **子流程定义** | 复杂流程是否已定义子流程边界？（避免主图过于臃肿） | ✅/❌ |

## 四、异常分支设计规范

### 4.1 异常分类标准

| 分类 | 触发条件 | 兜底策略 | 示例 |
|------|---------|---------|------|
| **参数异常** | 用户输入/前端传参不符合预期 | 前端拦截 + 提示用户修正 | 角色未选、角色非法 |
| **超时异常** | 下游服务响应超过阈值 | 自动重试 1 次 → 超时提示 + 重试按钮 | 数据查询超时 |
| **服务异常** | 下游服务不可用/返回 5xx | 降级到预设默认值/提示稍后再试 | Agent 服务不可用 |
| **安全异常** | 内容触发敏感词/安全策略 | 过滤/脱敏 + 标记 + 提示用户 | 内容安全不通过 |
| **网络异常** | 用户网络断开/弱网 | 检测网络状态 → 恢复后自动重试 | 网络断开 |

### 4.2 兜底处理模式

```
模式 A：中断式（Fail-Fast）
  触发 → 立即提示用户 → 结束/返回
  适用：参数异常

模式 B：重试式（Retry）
  触发 → 自动重试 N 次 → 成功/失败提示
  适用：超时异常

模式 C：降级式（Fallback）
  触发 → 返回默认值/缓存/模板 → 标记降级
  适用：服务异常

模式 D：补偿式（Compensate）
  触发 → 记录日志 → 告警 → 人工介入
  适用：严重异常（需人工处理）
```

## 五、追问机制设计规范

### 5.1 追问生命周期

```
触发追问 → 输入问题 → 意图识别 → 生成回复 → 展示回复
     ↑_________________________________________|
     │
退出追问 ← 点击关闭 / 超时 / 轮次超限 / 网络断开
```

### 5.2 追问退出条件

| 退出方式 | 触发条件 | 后续行为 |
|---------|---------|---------|
| 主动关闭 | 用户点击"关闭追问"按钮 | 收起追问区，保留历史记录 |
| 超时退出 | 用户 5 分钟未操作 | 自动收起追问区，保留历史记录 |
| 轮次超限 | 追问超过 50 轮 | 禁用追问输入框，提示"已达上限" |
| 网络断开 | 检测到网络异常 | 提示网络异常，恢复后可继续 |
| 页面关闭 | 用户关闭浏览器/标签页 | 前端 localStorage 保留，下次恢复 |

### 5.3 追问历史记录策略模板

| 维度 | 策略 |
|------|------|
| 前端保留 | `localStorage`，当前会话期间保留，刷新页面不丢失 |
| 后端保留 | 按用户 ID 保留最近 **7 天**（可配置） |
| 单条会话上限 | **50 轮**追问（超出后禁用追问输入框） |
| 自动清理 | 每日凌晨 3 点定时任务清理 **7 天前**的数据 |
| 脱敏日志 | 追问内容脱敏后保留 **90 天**，用于审计和模型优化 |
| 用户手动删除 | 支持用户手动清空当前会话的追问记录 |
| 追问内容入库 | **全量入库**（用于 badcase 分析和模型迭代） |

## 七、常见问题（FAQ）

**Q1：流程规划自审和蓝图自审的区别是什么？**

- 流程规划自审：检查"流程逻辑是否正确"（异常是否完整、追问是否闭环）
- 蓝图自审：检查"流程描述是否完整"（节点定义、连线覆盖、与 PRD 一致性）

**Q2：异常分支需要穷举所有情况吗？**

不需要穷举所有情况，但需要覆盖**高概率场景**和**高风险场景**。对于低概率场景，可以用兜底处理（如"其他异常 → 记录日志 → 提示用户"）。

**Q3：追问轮次 50 轮是否合理？**

需要根据业务场景评估：
- 如果每次追问都很简短，50 轮可能偏少
- 如果每次追问都包含复杂分析，50 轮可能偏多
- 建议：初始设置为 50 轮，上线后根据用户行为数据调整

**Q4：流程规划锁定后还能修改吗？**

原则上锁定后不再修改。如果必须修改，需要：
1. 说明修改原因
2. 重新执行流程规划逻辑自审
3. 重新生成蓝图和 drawio
4. 更新 PRD 相关章节
