## 2. 架构图 DrawIO 细则

### 2.1 继承 Mermaid 结构

| Mermaid 草稿 | DrawIO 终稿 |
|-------------|------------|
| `flowchart TB` + `subgraph` | 竖向 swimlane 容器 |
| `classDef appLayer` | `fillColor=#d1fae5, strokeColor=#10b981` |
| 内部节点（宽表/配置字段） | 矩形节点+分组框 |
| 4~6 根粗连线（汇报版） | 保留粗连线，新增子附图承载明细 |

### 2.2 汇报版 vs 落地版

| 版本 | 连线规则 | DrawIO 操作 |
|------|---------|------------|
| **汇报版（总图）** | 只画大模块间 4~6 根粗连线 | 主图中只保留汇总连线，内部节点用分组框罗列 |
| **落地版（子图）** | 全量细节连线 | 新增附属子图，画 AI 中台 → 各宽表的明细调用关系 |

### 2.3 排版规范

| 项 | 标准 |
|----|------|
| 间距 | 同级节点水平间距 ≥ 200px，垂直间距 ≥ 150px |
| 框大小 | 同层级节点宽高统一（宽 120px × 高 60px） |
| 字体 | 中文：PingFang SC / Microsoft YaHei，12px |
| 标题 | 层级名称 14px 加粗，模块名称 12px |
| 图例 | 右上角独立图例框，标注配色含义和虚实线定义 |

```
┌──────────┬──────────┬──────────┬──────────┐
│ 用户端    │ 前端页面  │ AI Agent层│ Ontology │
├──────────┼──────────┼──────────┼──────────┤
│ 发起查询   │          │          │          │
│     │      │          │          │          │
│     └──────► 意图理解  │          │          │
│          │     │      │          │          │
│          │     └──────► 数据检索 │          │
│          │          │     │      │          │
│          │          │     └──────► 字段查询 │
│          │          │          │     │      │
│          │          │          │     └──►  │
│          │          │     ◄────── 数据回传  │
│     ◄────── 输出报告  │          │          │
│ 查看报告   │          │          │          │
└──────────┴──────────┴──────────┴──────────┘
```

### 3.2 DrawIO 泳道图操作规范

| 项 | 标准 |
|----|------|
| 泳道列 | 标准竖线分割，顶部表头标注系统名称，字号 14px 加粗 |
| 流程走向 | 从上到下，主干居中，分支向左右延展 |
| 实线 | 主动同步调用（跨泳道） |
| 虚线 | 数据回传 / 异步回调 / 异常降级（跨泳道） |
| 总图 | 只画跨系统粗粒度箭头，删除 AI 内部→各宽表的细线 |
| 子图 | 新增「数据检索明细泳道图」，画 AI 中台 → 各宽表/决策引擎的全量调用 |

### 附录 C3：连线专项绘制细则（DrawIO 泳道专用）

> **铁律口诀：主动向外请求 = 实线，结果回来 / 异常兜底 = 虚线**

#### C3.1 线型铁则（全文档和架构图共用一套标准）

| 线型 | DrawIO 属性 | 适用场景 |
|------|-----------|---------|
| **实线** `dashed=0` | `edgeStyle=orthogonalEdgeStyle;rounded=1;dashed=0` | ① 用户 / 上层系统**主动发起请求**（前端调 AI、AI 调 Ontology、AI 调 SR 数据库）<br>② **主动查询**、**主动下发指令**（所有同步主动调用 = 实线） |
| **虚线** `dashed=1` | `edgeStyle=orthogonalEdgeStyle;rounded=1;dashed=1` | ① **下层数据向上返回结果**（数据库→AI、AI→前端页面）<br>② **异常链路**：超时、失败、降级兜底分支<br>③ **异步回调**、延迟推送 |

**常见错误纠正对照表：**

| ❌ 错误 | ✅ 正确 | 原因 |
|---------|---------|------|
| 前端→AI「解析请求」画虚线 | 实线 | 前端主动调用 AI = 实线 |
| AI→Ontology「语义查询」画虚线 | 实线 | AI 主动拉取配置 = 实线 |
| AI→SR「SQL查询」画虚线 | 实线 | AI 主动查库 = 实线 |
| SR→AI 数据回传画实线 | 虚线 | 数据回传 = 虚线 |
| 超时→降级→概览输出画实线 | 虚线 | 异常兜底链路 = 虚线 |
| LLM失败→降级输出画实线 | 虚线 | 异常回调 = 虚线 |
| 前端→用户「展示报告」画实线 | 虚线 | 数据回传给用户 = 虚线 |

#### C3.2 连线文字约束

| 规则 | 说明 |
|------|------|
| ❌ **严禁**在连线线段上书写任何**业务说明** | 如「SQL 查询」「查询请求」「解析请求」等——这是业务操作描述，必须收纳在节点名称中 |
| ✅ 所有业务描述文字**统一写入对应节点框内** | 通过节点名称表达含义，连线只负责表达流向 |
| ✅ **菱形分支连线**可标注**流转条件** | 如「正常」「降级」「追问」——这是 BPMN 分支条件，不是业务操作描述 |

**连线 value 属性规则：**
- 普通连线：`value=""`（空值，禁止写业务说明）
- 菱形出线：`value="正常"` 或 `value="降级"` 或 `value="追问"`（仅写流转条件）

#### C3.3 箭头形态约束

| 规则 | 说明 |
|------|------|
| ✅ 所有流转**单向箭头** | 每条连线只有一个方向箭头 |
| ✅ 业务双向交互**必须拆分** | 正向调用实线 + 结果回传虚线 = 两根独立连线 |
| ❌ **禁止**双向箭头 | BPMN 规范不允许单根双向箭头 |
| ❌ **禁止**无箭头悬浮连线 | 所有连线必须有明确的箭头方向 |

**示例：追问页双向交互拆分：**
- 用户→前端：实线（主动发起追问）
- 前端→用户：虚线（结果回传展示）

#### C3.4 连线路由约束

| 规则 | 说明 |
|------|------|
| ✅ 跨泳道连线**走两列中间空白走廊** | 不从节点方框内部穿线 |
| ✅ 主干流程连线**竖直 / 水平正交走线** | 减少折线，优先正交 |
| ❌ **禁止**横穿 / 斜穿任意节点方框 | 连线必须绕行，使用 waypoints 控制路由 |
| ✅ 回传虚线**走与主动调用不同的走廊** | 避免实线虚线在同一路径重叠 |

**DrawIO 路由实现：**
- 使用 `exitX/exitY/entryX/entryY` 精确控制出入口
- 使用 `<mxPoint>` waypoints 控制中间折点
- 同一节点多条出线时，分散出口（`exitX=0.25/0.5/0.75`）

#### C3.5 分支连线约束

| 规则 | 说明 |
|------|------|
| ✅ 菱形判断节点每条出边**必须标注分支文案** | 如：`正常`、`降级`、`追问`、`超时` |
| ❌ **禁止**菱形出线空白无标注 | 无法区分流转分支 |
| ✅ 全业务流程**首尾必须有起止节点** | 开始椭圆 + 结束椭圆 |
| ✅ 收尾连线**统一汇入结束椭圆** | 所有流程最终闭环收束 |

### 附录 C3.5：通用前置对齐总则（三类图纸共用）

> **适用**：BPMN 泳道图、普通流程图、系统分层架构图，三类图纸共用同一套对齐原则。

| # | 原则 | 说明 | 示例 |
|---|------|------|------|
| 1 | **成对节点等高** | 同一次请求触发、上下游跨容器 / 跨泳道的成对节点，水平等高对齐（y 坐标相同） | 用户【发起查询】↔ 前端【接收查询请求】同高并排 |
| 2 | **并列分支并排** | 同一个父节点拆分出多个并列分支子节点：横向并排、同一水平线 | 角色适配分出【正常 / 降级】两个分支同高并排 |
| 3 | **串行步骤纵向** | 串行先后执行的步骤：垂直上下排布，不并排 | AI 泳道：意图理解→数据检索→语义解读，纵向串行 |
| 4 | **锚点统一** | 所有节点统一：矩形左右中点为连线锚点（左进右出） | 跨泳道调用从右侧中点出，从左侧中点入 |

#### C4.5 等高对齐规则（修订版）

```markdown
1. 跨泳道同步调用：调用节点与被调用节点必须等高（y坐标一致）
2. 同泳道串行步骤：纵向排列，不等高

以下场景不适用等高对齐，必须错位排布：
   a) 一个节点同时发起多个并行跨系统调用（见 C3.9）
   b) 多个节点同时调用同一个下游节点
   c) 循环交互的回流链路
```

#### C4.1 跨泳道成对节点规则（🚨 最重）

> **A 泳道节点主动调用 B 泳道节点，调用节点 ↔ 被调用节点 y 坐标完全一致、同一条水平线。**

| 场景 | 规则 | 示例 |
|------|------|------|
| **跨泳道主动调用** | 发起节点与接收节点**强制水平等高** | 用户【发起查询】↔ 前端【接收查询】同高；前端【解析完毕】↔ AI【意图理解】同高 |
| **回流数据** | 返回节点和上游触发节点**尽量等高** | SR 宽表查询结果 → AI 数据检索，尽量等高对齐 |
| **同节点多路并发** | 所有分支结果节点**强制水平等高** | AI 同时查「用户宽表、授信宽表」→ 两个 SR 节点等高并排 |
| **菱形多分支结果** | 分支落地节点**强制水平等高** | 菱形角色适配 → 正常输出、降级输出，两个落地节点等高 |

| ❌ 不用等高 | 场景 |
|-------------|------|
| ❌ 时序串行节点 | 同一泳道内先后步骤：AI 意图理解→数据检索→语义解读，**纵向上下排布，不等高** |
| ❌ 流程有先后顺序 | 一步做完再走下一步：纵向错开，不等高 |

**极简口诀：**
- 一问一答、一调一收 → 跨列**必须**等高（泳道最常用）
- 一源多分、并发并行 → 同层**必须**等高
- 先后执行、步骤串行 → 上下错开**不等高**

### 附录 C4-A：系统分层架构图排布规则

#### 同层级内部
- 同一分组内：有先后依赖的模块**竖向上下排布**；平级独立模块**横向并排、等高对齐**
- 不同分组：分组标题统一在同一水平线

#### 跨层级节点对齐
- 上层模块向下调用下层模块，两个模块**尽量垂直对齐**（竖向对齐，不是等高），方便竖直连线，减少斜线
- 例：应用层页面 → 中台能力，上下尽量居中对齐

#### 数据配置子分组（Ontology / 元数据）
- 内嵌在数据层分组内部，同配置集的多个配置项**横向并排**

### 附录 C5：连线布线硬性规范（DrawIO 泳道专用）

#### C5.1 正交走线铁则

| 规则 | 说明 |
|------|------|
| ✅ 所有连线**优先正交横竖线** | 禁止 45° 斜线跨泳道 |
| ✅ 跨泳道连线**走水平走廊** | 只在泳道间隙做水平平移 |
| ❌ **禁止**斜穿跨列连线 | AI→Ontology、AI→SR 多条斜穿 = 丑的核心来源 |

**正交走线两步法：**
1. **先竖后横**：从节点右侧出口出发 → 垂直下降到目标 y → 水平穿越走廊 → 垂直上升进入目标节点左侧入口
2. **先横后竖**：从节点右侧出口 → 水平穿越走廊 → 垂直上下到目标 y → 进入目标节点左侧入口

#### C5.2 锚点统一规则（🚨 全图统一）

| 方向 | 出线锚点 | 入线锚点 | DrawIO 属性 |
|------|---------|---------|------------|
| 跨泳道右→左（实线主动调用） | `exitX=1;exitY=0.5`（右侧中点） | `entryX=0;entryY=0.5`（左侧中点） | 标准值 |
| 跨泳道左→右（虚线数据回传） | `exitX=0;exitY=0.5`（左侧中点） | `entryX=1;entryY=0.5`（右侧中点） | 标准值 |
| 同列向下（实线内部流转） | `exitX=0.5;exitY=1`（底部中点） | `entryX=0.5;entryY=0`（顶部中点） | 标准值 |
| 同列向上（虚线异常/降级） | `exitX=0.5;exitY=0`（顶部中点） | `entryX=0.5;entryY=1`（底部中点） | 标准值 |
| 菱形多出口分散 | `exitX=1;exitY=0.25/0.5/0.75` | 标准入口 | 分散出口 |

| ❌ 禁止 | 说明 |
|---------|------|
| ❌ 连线连方框顶部又连侧边又连底部 | 锚点不统一 = 视觉混乱 |
| ❌ 出线从节点左上方出发 | 应统一右侧中点出 |
| ❌ 入线从节点底部接入 | 应统一左侧中点入 |

#### C5.3 绕短线就近原则

| 规则 | 说明 |
|------|------|
| ✅ 回传虚线**就近走横向走廊** | LLM→前端、SR超时→Agent，走最近的泳道间隙水平穿越 |
| ❌ **禁止**绕远路横穿 3~4 个泳道 | 降级链路、追问回流线必须就近路由 |
| ✅ 多条跨泳道线**共用走廊不同 y 层** | 避免重叠，每条线占不同的 y 坐标 |

#### C5.4 串行调用节点布局与回传规则（强制）

> **场景**：多个节点串行执行（A→B→C→D），每个节点都可能需要把结果回传给上游父节点。

**第一步：节点水平等高排布**
```
水平方向从左到右依次摆放 N 个节点，必须完全水平等高对齐（y 坐标完全一致）
左：父节点【数据检索】
中：子节点1【字段含义查询】（y 错位：向上偏移 ≥ 40px）
右：子节点2【宽表数据查询】（y 错位：向下偏移 ≥ 40px）
```

**第二步：预留回传路由走廊**
```
在所有节点正上方预留 2 个网格高度的空白区域（专属回传路由走廊）
走廊内不放任何节点
```

**第三步：正向调用线（实线，dashed=0）**
```
从【父节点】右侧中点出发 → 水平向右实线 → 连接到【子节点1】左侧中点
从【子节点1】右侧中点出发 → 水平向右实线 → 连接到【子节点2】左侧中点
效果：所有正向调用线在同一条水平线上，笔直向右，不拐弯
```

**第四步：回传虚线（虚线，dashed=1）**
```
第一条（最右侧回传）：
  从【子节点N】右侧中点出发
  → 垂直向上 2 个网格，进入上方回传走廊
  → 水平向左画直线，一直画到【父节点】正上方
  → 垂直向下 2 个网格，连接到【父节点】右侧中点
  → 整条线设为虚线

第二条（中间回传）：
  从【子节点1】右侧中点出发
  → 垂直向上 2 个网格，进入同一条上方回传走廊
  → 水平向左画直线
  → 垂直向下 2 个网格，连接到【父节点】右侧中点
  → 整条线设为虚线
```

**第五步：扩展规则（N 个串行节点）**
```
无论多少个串行节点，所有回传线都走同一条上方走廊：
  每个节点的回传线都先向上拐入走廊
  然后全部水平向左走
  最后统一向下拐入最左侧的父节点
效果：永远不会出现连线交叉
```

**禁止事项（写死）**：
| # | 禁止 | 说明 |
|---|------|------|
| 1 | 正向线和回传线在同一水平线 | 必须上下分离 |
| 2 | 回传线直接横向向左拉 | 必须先上/下拐入走廊 |
| 3 | 同一节点的回传线从不同侧进出 | 统一从右侧出、右侧入 |
| 4 | 多个节点的回传线交叉 | 同一条走廊内平行不交叉 |

### 附录 C6：节点连通性强制约束

| 规则 | 说明 |
|------|------|
| ✅ **所有流程节点必须有上游入线 + 下游出线** | 起止节点除外 |

### 附录 C3.7：串行调用 + 结果回流绘制规则（强制）

```markdown
1. 正向调用（实线，dashed=0）：
   - 统一从左向右横向走线
   - 禁止向左或双向

2. 结果回传（虚线，dashed=1）：
   - 必须走节点上方或下方的空白路由走廊
   - 禁止直接横向向左拉虚线
   - 多节点串行时，结果走同一条走廊回流，避免交叉

3. 循环交互：
   - 回流线统一走节点下方，形成 U 型闭环
   - 禁止横向直接回拉形成视觉双向箭头
```

### 附录 C3.8：禁止视觉双向箭头判定规则（强制）

```markdown
即使业务逻辑是双向交互，也不允许出现"左→右实线 + 右→左虚线"在同一水平线上并排的情况。
必须通过上下走廊分离两条线的走线位置。
```

### 附录 C3.9：一对多并行跨系统调用规则（强制）

```markdown
1. 核心原则：打破等高，错位排布
   - 一对一同步调用：调用方与被调用方强制等高
   - 一对多并行调用：所有被调用节点必须上下错位，禁止全部和调用方等高
   - 错位间距：相邻被调用节点垂直间距≥40px

2. 分层走线法（必用）：
   - 第1个并行调用：走节点上方走廊
   - 第2个并行调用：走节点下方走廊
   - 第3个及以上：依次在上下走廊之间新增独立水平走廊
   - 所有调用线和回传线必须走专属走廊，禁止在同一水平线上重叠

3. 并行汇合规则：
   - 所有并行调用的结果都回传给同一个父节点时，回传线必须走各自对应的走廊，最后在父节点处汇合
   - 禁止直接横向回拉，必须先拐入走廊再横向走线

4. 超过3个并行调用：使用汇总节点法（见附录 C3.9.1）
```

#### 附录 C3.9.1：超过3个并行调用的汇总节点法

```markdown
在父节点右侧新增一个"并行调用汇总"节点（和父节点等高）
父节点只画一条实线指向汇总节点
汇总节点再分别向下错位引出多条调用线到各个被调用系统
所有回传线都先回到汇总节点，再统一回传给父节点

示例：
数据检索 → 并行调用汇总
  ↓ 汇总节点向下错位引出3条线
  ├→ 系统A
  ├→ 系统B
  └→ 系统C
  ↓ 结果都回到汇总节点
并行调用汇总 → 数据检索
```

### C9.1 生成策略：只画节点，不画连线

|------|------|------|
| **架构图** | 只生成模块节点，不生成连线 | 架构图连线应由用户根据实际交互关系自行添加，AI 生成的连线易出现交叉、穿越、弯曲等问题 |
| **流程图** | 只生成步骤节点（含 Agent 内部展开），不生成连线 | 流程图连线路径复杂，AI 生成的连线易出现交叉、异常分支线过多、弯曲等问题 |
| **泳道图** | 只生成节点，不生成连线 | 泳道图跨泳道连线极易出现穿越、交叉、不等高等问题，应由用户手动调整 |

**执行方式**：
```xml
<!-- 只定义 mxCell vertex（节点），不定义 mxCell edge（连线） -->
<mxCell id="node1" ... vertex="1" />  <!-- ✅ 生成 -->
<mxCell id="edge1" ... edge="1" />    <!-- ❌ 不生成 -->
```

### C9.2 Agent 内部流程展开规则

**禁止做法**：
```xml
<!-- ❌ 错误：用"子流程"节点+文字说明代替展开 -->
<mxCell value="Report Agent&#xa;报告生成（子流程）" ...>
<!-- 下方文字说明：角色校验→确定查询范围→... -->
```

**正确做法**：
```xml
<!-- ✅ 正确：在流程图中将 Agent 内部步骤作为独立节点画出 -->
<mxCell value="角色校验" ... />
<mxCell value="确定查询范围" ... />
<mxCell value="数据检索" ... />
<!-- ... 每个步骤都是独立节点 ... -->
```

**展开规则**：
- Report Agent：角色校验 → 确定查询范围 → 数据检索 → SQL 生成 → 宽表查询 → 语义解读 → 角色适配 → 报告生成 → 内容过滤（9 个节点）
- FollowUp Agent：意图识别 → 追问回复生成（2 个节点）
- 每个节点独立，不画连线

### C9.3 泳道图间隙规则

| 项 | 规则 |
|----|------|
| **泳道列** | 有颜色背景（与系统/角色对应） |
| **泳道间隙** | **留白**，无背景色、无填充 |
| **泳道名称** | 顶部水平表头（标准 BPMN），不在侧边旋转 |

**错误示例**：
```xml
<!-- ❌ 错误：泳道间隙有背景色 -->
<mxCell ... fillColor=#f8fafc; ...>  <!-- 间隙不应有填充 -->

<!-- ❌ 错误：泳道名称在侧边旋转 -->
<text ... rotation=90; ...>  <!-- 不应旋转 -->
```

**正确示例**：
```xml
<!-- ✅ 正确：泳道列有颜色，间隙留白 -->
<mxCell ... fillColor=#d1fae5; ...>  <!-- 泳道列有颜色 -->
<!-- 间隙无 mxCell，天然留白 -->

<!-- ✅ 正确：泳道名称在顶部水平显示 -->
<text ... align=center; ...>用户端</text>
```

### C9.4 异常处理表示规则

| 场景 | 表示方式 | 禁止 |
|------|---------|------|
| **异常分支** | 在相关节点旁用文字说明框标注 | ❌ 禁止画额外的异常分支线和异常节点 |
| **异常说明** | 红色/浅色背景的文字框，写明异常条件和处理方式 | ❌ 禁止用菱形→异常节点→返回线的方式 |

**正确示例**：
```xml
<!-- ✅ 正确：用文字说明框标注异常 -->
<mxCell value="异常：角色未选/非法&#xa;处理：提示\"请先选择角色\""
         style="text;html=1;fillColor=#fef2f2;strokeColor=#fee2e2;rounded=1;"
         ... />
```

### C9.6 自检规则（强制更新）

#### 自检流程（修正版）

```
Step 1: 生成 drawio XML → 写入文件
Step 2: 导出 PNG → draw.io CLI 导出
Step 3: 🔴 读取 PNG 进行视觉验证（AI 必须读取图片）
Step 4: 逐项对照自检清单
Step 5: 修复问题 → 回到 Step 1
Step 6: 再导出 PNG → 确认无问题 → 交付
```

| # | 检查项 | 标准 | 判定 |

| 7 | **泳道名称位置** | 顶部水平表头，不在侧边旋转 | ✅/❌ |
