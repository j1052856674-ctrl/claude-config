# 需求评审 Skill

## 定位

PRD 流程的前置步骤。输入用户需求描述，输出需求评审报告 + 需求卡 + 关键决策点。

**核心价值**：帮助用户把需求想清楚，避免"拍脑袋"做产品。

**设计原则**：
- **渐进式追问**：分 3 轮对话逐步深入，而非一次性抛出所有问题。降低用户回答负担，每轮确认理解后再进入下一轮。
- **用户旅程视角**：第 3 轮追问聚焦用户旅程（进入→动作→反馈→结果），补全从用户视角出发的需求细节。
- **需求卡验证**：所有追问完成后，生成一页纸「需求卡」让用户验证 AI 理解是否准确，确认后再进入流程规划。

## 执行流程（10 步）

### Step 1: 复述理解

复述用户对需求的理解，确认是否正确。**这一步不追问、不推荐模式**，只做理解确认。

```
我理解您的需求是：
- 产品类型: [智能推断]
- 核心诉求: [从需求描述提取]
- 目标用户: [推断]
- 使用场景: [推断]

请确认或补充。如果理解有偏差，请直接纠正。
```

---

### Step 2: 第1轮追问 — Who + What（目标用户 & 核心功能）

用户确认复述后，追问用户画像和核心功能。**只问这两个维度，不涉及 Why/How/旅程**。

| 维度 | 问题 | 作用 |
|------|------|------|
| **Who** | 目标用户是谁？具体是什么角色/岗位？ | 明确给谁用，用户画像 |
| **What** | 核心功能是什么？用户用这个产品完成什么任务？ | 明确产品边界 |

**追问话术**：
```
先聊两个最核心的问题：

1. 目标用户是谁？（具体角色/岗位，比如"风控策略分析师"而非泛泛的"运营人员"）
2. 核心功能是什么？（用户用这个产品完成什么任务？一句话概括即可）

这两个问题确认后，我们再深入聊使用场景。
```

---

### Step 3: 第2轮追问 — Why + How（业务价值 & 使用方式）

Step 2 确认后，追问业务价值和使用方式。

| 维度 | 问题 | 作用 |
|------|------|------|
| **Why** | 为什么要做这个？解决什么痛点？不做会怎样？ | 明确业务价值，避免伪需求 |
| **How** | 用户怎么用？大概的使用流程是怎样的？ | 明确交互方式雏形 |

**追问话术**：
```
用户画像和核心功能确认了。接下来聊聊价值和用法：

3. 为什么要做这个产品？当前有什么痛点？如果不做，用户现在怎么解决这个问题？
4. 用户大概会怎么使用它？（不需要很细，讲个大致的使用流程就行）
```

---

### Step 4: 第3轮追问 — 用户旅程补全（进入→动作→反馈→结果）

Step 3 确认后，以用户视角补全使用旅程。**聚焦用户体验，而非系统实现**。

| 旅程节点 | 问题 | 作用 |
|---------|------|------|
| **进入** | 用户怎么进入产品？从哪里来？看到什么？ | 明确入口和初始状态 |
| **每一步动作** | 用户每一步做什么操作？期望完成什么？ | 明确核心交互 |
| **每一步反馈** | 系统给用户什么反馈？用户怎么知道操作成功了？ | 明确体验闭环 |
| **最终结果** | 最终呈现什么结果？用户怎么知道任务完成了？ | 明确成功标准 |

**追问话术**：
```
价值和用法清楚了。现在我们从用户视角走一遍完整流程：

5. 用户从哪里进入产品？打开后第一眼看到什么？
6. 用户接下来每一步做什么操作？期望完成什么？
7. 每一步操作后，系统给用户什么反馈？（比如：提示、结果展示、状态变化）
8. 最后用户得到什么结果？怎么知道任务完成了？

不需要很专业的产品术语，用大白话描述就行。比如"用户点一个按钮，系统查数据，然后把结果展示出来"。
```

**输出用户旅程摘要**（确认后写入评审报告）：

```markdown
## 用户旅程摘要

| 阶段 | 用户动作 | 系统反馈 | 用户情绪 |
|------|---------|---------|---------|
| 进入 | [用户怎么进入/看到什么] | [初始界面/状态] | [预期情绪] |
| 操作 | [每一步做什么] | [每一步的反馈] | [预期情绪] |
| 完成 | [怎么知道做完了] | [最终结果呈现] | [预期情绪] |
```

---

### Step 5: 目标用户识别

汇总前 4 步收集的信息，输出结构化用户画像：

```markdown
## 目标用户

| 维度 | 描述 |
|------|------|
| 角色 | [具体角色/岗位] |
| 痛点 | 他们当前面临的痛点是什么？ |
| 需求 | 他们需要什么？ |
| 使用频率 | 高频 / 低频 |
| 使用场景 | [从 Step 4 提取] |
```

---

### Step 6: 价值评估

```markdown
## 价值评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 业务价值 | 高/中/低 | 解决了什么业务痛点？能带来的收益？ |
| 用户价值 | 高/中/低 | 目标用户有多大？用户需求的迫切程度？ |
| 技术可行性 | 高/中/低 | 技术难度如何？是否可实现？ |
| 竞争差异化 | 有/无 | 与竞品相比有什么优势？ |
| 投入产出比 | 高/中/低 | 投入成本 vs 预期收益 |
```

---

### Step 7: 合理性检查

检查需求是否合理：

```markdown
## 合理性检查

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 需求是否明确 | ✅/❌ | |
| 目标用户是否清晰 | ✅/❌ | |
| 使用场景是否完整 | ✅/❌ | |
| 用户旅程是否闭环 | ✅/❌ | [从 Step 4 验证] |
| 技术是否可行 | ✅/❌ | |
| 是否有竞品参考 | ✅/❌ | |
```

---

### Step 8: 竞品参考（可选）

快速搜索 1-2 个类似产品，提供参考。

```markdown
## 竞品参考

| 产品 | 功能 | 优点 | 缺点 |
|------|------|------|------|
| 竞品A | ... | ... | ... |
| 竞品B | ... | ... | ... |
```

---

### Step 9: 关键决策点输出

输出需要用户确认的关键决策点：

```markdown
## 关键决策点（请确认）

| # | 决策点 | 默认方案 | 备选方案 | 影响 |
|---|--------|---------|---------|------|
| 1 | [决策点名称] | [推荐方案] | [备选] | [影响说明] |
| 2 | ... | ... | ... | ... |

请确认以上决策点，确认后生成需求卡。
```

---

### Step 10: 生成需求卡（一页纸验证）

> ⚠️ **这是需求理解验证的关键节点**。在进入流程规划前，必须生成需求卡让用户验证 AI 的理解是否准确。

需求卡是一页纸摘要，只包含最核心的信息。用户验证通过后，需求卡作为后续 PRD 生成的"理解基线"。

```markdown
# [项目名] — 需求卡 v1

## 产品一句话
[一句话说清这个产品是什么、给谁用、解决什么问题]

## 用户画像
- **角色**：[具体角色]
- **痛点**：[核心痛点]
- **目标**：[用户想达成什么]

## 核心场景
[描述一个最典型的使用场景，2-3 句话]

## 用户旅程（概要）
1. **进入** → [用户怎么进来，看到什么]
2. **操作** → [用户做什么]
3. **反馈** → [系统给什么反馈]
4. **结果** → [最终呈现什么]

## 期望结果
- 用户用完后得到什么？
- 怎么判断"做对了"？

---
> 💡 以上理解对吗？如果有偏差请指出，我调整后再进入下一步。
> 确认无误后，将进入"流程规划"阶段。
```

---

## 需求卡验证规则

1. **必须生成**：需求卡是必出产物，不可跳过
2. **用户确认**：用户明确确认后，需求卡锁定，作为后续 PRD 的"理解基线"
3. **允许修正**：用户指出偏差时，回到对应步骤修正信息，更新需求卡版本号（v2, v3...）
4. **修正上限**：同一维度修正不超过 2 次，超过则标记为"[待确认]"并继续

## 渐进追问速查

| 轮次 | 追问维度 | 核心问题 | 确认后才进入 |
|------|---------|---------|------------|
| 第1轮 | Who + What | 目标用户是谁？核心功能是什么？ | 第2轮 |
| 第2轮 | Why + How | 为什么做？用户怎么用？ | 第3轮 |
| 第3轮 | 用户旅程 | 进入→动作→反馈→结果 | 目标用户识别 |

**降级规则**：如果某轮用户表示"没想清楚"或"不确定"，标注 `[待确认]`、跳过该轮追问、继续后续步骤。不要在某轮反复追问。
