# 契约驱动开发流程

> 本文件定义了从PRD到开发执行的闭环流程。
> 核心思路：PRD是开发契约，开发过程严格参照PRD规格，完成后对照PRD验证实现合规性。

---

## 一、流程总览

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Phase 1        Phase 2        Phase 3        Phase 4      │
│  写PRD     →    契约化     →    契约开发   →    契约验证    │
│                                                             │
│  产出PRD文件     产出TaskList     逐Task执行     合规报告    │
│  完整性检查      + contract.md    参照PRD规格    逐项对照    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 二、Phase 1：写 PRD

> 由 prd-v3 skill 执行。

### 产出物
- `prd/[项目名].md` — 完整的PRD文档
- 完整性检查结果（对照对应Checklist）

### 质量门禁
- 所有必填项（带*）都有实质内容
- 交互规格章节逐字段描述（不是笼统描述）
- 风控项目包含安全合规章节

### 判定
- ✅ 全部必填项满足 → 进入 Phase 2
- ⚠️ 有缺失项 → 先补充再进入
- ❌ 3+项缺失 → 回退重写

---

## 三、Phase 2：契约化 — PRD → 开发任务

> 当用户说"基于这个PRD开发"或"开始实现"时执行。

### 步骤

#### 1. 读取PRD文件
```
使用 Read 工具读取 prd/[项目名].md
```

#### 2. 解析为开发任务

将PRD的每个功能模块/交互元素解析为独立Task：

**解析规则**：
- 每个页面 → 1个Task（包含该页面所有交互规格）
- 每个独立功能模块 → 1个Task
- 每个新增数据字段批次 → 1个Task
- 每个接口定义 → 1个Task

**Task描述必须包含**：
- 对应PRD章节编号（如 §2.5）
- **完整规格原文**（不是摘要！直接引用PRD原文）
- 验收标准（从PRD规格中提取的可验证点）

**示例Task**：
```
Task T1: 实现对话输入框
来源：PRD §2.5 "对话输入框"

规格原文（从PRD直接引用）：
"支持中英文、数字、常用符号及多行文本输入，内容超出高度时自动纵向延展，
不出现横向滚动条，纵向超出后可通过鼠标滚轮滚动查看。单轮最大2000字符，
超出时输入框边框高亮红色+弹出提示"输入内容超出长度限制（最大2000字符）"，
禁止继续输入。Ctrl+V支持粘贴，Enter/Ctrl+Enter发送，Shift+Enter换行。
默认提示"请用自然语言描述需求..."，点击后提示消失，输入清空后重新显示。
AI生成中时输入框置灰不可编辑。"

验收标准：
1. 输入框支持多行文本输入，自动纵向延展
2. 超出2000字符时红色边框+提示弹出
3. Ctrl+V粘贴、Enter发送、Shift+Enter换行均可工作
4. 默认提示文案正确显示/消失
5. AI生成中输入框置灰不可编辑
```

#### 3. 创建契约文件

保存 `prd/[项目名]-contract.md`，记录PRD章节与Task的映射关系：

```markdown
# 开发契约 — [项目名]

## 任务映射表

| Task ID | PRD章节 | 任务内容 | 优先级 | 状态 |
|---|---|---|---|---|
| T1 | §2.5 对话输入框 | 实现输入框交互规格 | P0 | 未开始 |
| T2 | §2.5 历史对话 | 实现历史对话列表 | P0 | 未开始 |
| T3 | §2.5 文件上传 | 实现文件上传功能 | P1 | 未开始 |
| T4 | §2.5 对话辅助按钮 | 实现复制/编辑/生成/点赞/踩 | P1 | 未开始 |
| T5 | §2.6 话术设计 | 实现智能体话术逻辑 | P0 | 未开始 |
| T6 | §2.7 安全合规 | 实现脱敏/权限/审计 | P0 | 未开始 |

## 验收标准汇总

[每个Task的验收标准列表]
```

## 四、Phase 3：契约驱动开发

> 逐Task执行开发，每个Task必须参照对应的PRD章节。

### 开发执行规则

1. **开始Task前**：读取对应的PRD章节原文
2. **设计决策时**：优先参照PRD规格，不自由发挥
   - 如果PRD规定了按钮名称 → 使用规定名称，不自创
   - 如果PRD规定了交互行为 → 按规定实现，不简化
   - 如果PRD规定了提示文案 → 使用规定文案，不改写
3. **PRD未规定时**：
   - 采用行业最佳实践
   - 在契约文件中记录"⚠️ PRD未规定，采用XX方案"
   - 通知用户确认
4. **发现PRD规格不合理时**：
   - 不擅自修改实现
   - 标注"⚠️ 待确认：PRD规格XX可能不合理，建议XX"
   - 通知用户决策
5. **Task完成时**：更新契约文件中该Task状态为"已完成"

### 关键原则

> **核心原则：PRD是契约，不是参考。**
> 开发人员（包括AI）必须严格按照PRD规格实现，
> 不可以因为"我觉得更好的方式"而偏离PRD规定。
> 如果要改PRD，必须通过变更记录流程（§7），不能暗改。

---

## 五、Phase 4：契约验证 — 对照PRD检查实现

> 所有Task完成后，对照PRD逐项验证实现合规性。

### 验证步骤

#### 1. 逐章节对照PRD

读取PRD文件和契约文件，逐章节对照检查：

```markdown
## 验证矩阵

| PRD需求项 | 规格描述 | 实现状态 | 是否符合 | 差异说明 |
|---|---|---|---|---|
| 输入框最大字符 | 2000字符，超出红色边框+提示 | ✅ 已实现 | ✅ 符合 | - |
| 发送快捷键 | Enter/Ctrl+Enter发送，Shift+Enter换行 | ✅ 已实现 | ⚠️ 部分符合 | Ctrl+Enter未实现，只支持Enter |
| 文件上传格式 | Excel/CSV/TXT，≤5个 | ✅ 已实现 | ✅ 符合 | - |
| 复制按钮 | 仅复制纯文本，成功提示2秒消失 | ✅ 已实现 | ❌ 不符合 | 提示未自动消失 |
| 点踩反馈 | 展开6个原因选项+其他自定义 | ✅ 已实现 | ✅ 符合 | - |
```

#### 2. 分类统计

| 类别 | 定义 | 处理方式 |
|---|---|---|
| ✅ 完全符合 | 实现与PRD规格完全一致 | 无需处理 |
| ⚠️ 部分符合 | 实现了核心功能但有细节差异 | 记录差异 → 评估是否需要修复 |
| ❌ 不符合 | 实现与PRD规格有重要差异 | 必须修复 → 修复后重新验证 |
| 📝 PRD未规定 | PRD未规定但自行补充了功能 | 记录补充内容 → 用户确认是否保留 |

#### 3. 输出合规报告

```markdown
# 合规报告 — [项目名]

## 概要
- ✅ 完全符合：[N]项
- ⚠️ 部分符合：[N]项 → [差异列表]
- ❌ 不符合：[N]项 → [差异列表]
- 📝 PRD未规定，自行补充：[N]项 → [补充内容列表]

## 需修复项
[列出所有❌不符合项的具体差异和修复方案]

## 需确认项
[列出所有⚠️部分符合和📝自行补充项，等待用户决策]
```

#### 4. 处理闭环

- ❌ 不符合 → 修复实现 → 重新验证该项
- ⚠️ 部分符合 → 用户决定是否修复
- 📝 自行补充 → 用户决定保留或移除
- 全部 ✅ → 契约验证通过 → 项目完成

---

## 六、异常处理

| 异常情况 | 处理方式 |
|---|---|
| PRD规格有矛盾 | 标注矛盾点 → 用户决定取哪个版本 |
| PRD规格与技术实现不可行 | 标注不可行原因 → 提出替代方案 → 用户确认 |
| 开发中发现新需求 | 不暗加 → 记录到PRD变更记录（§7） → 用户确认后再开发 |
| 多Task间依赖冲突 | 在契约文件中标注依赖关系 → 按依赖顺序执行 |

## 八、快速使用指南

### 从PRD到开发的一句指令

```
用户：基于 prd/清退策略Agent.md 开始开发

系统自动执行：
1. Read prd/清退策略Agent.md → 解析PRD内容
2. 创建 prd/清退策略Agent-contract.md → 契约文件

4. 逐Task执行 → 参照PRD规格实现
5. 全部完成后 → 对照PRD验证合规性 → 输出合规报告
```
