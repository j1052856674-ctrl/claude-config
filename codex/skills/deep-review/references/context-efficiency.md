# 上下文效率检查清单

> Phase 2 执行时加载。目标：评审过程和评审对象都要最小化不必要的 token 消耗。

---

## 一、评审对象的指令密度

- [ ] **核心指令占比 >50%？** 排除教程、示例、模板、推导过程后的纯检查项/判定标准占比
- [ ] **按场景裁剪加载？** 还是一股脑全载？（反例：评审纯文档 Skill 不需要 CLI 验证框架）
- [ ] **是否存在"教科书随船携带"？** 每次加载大量用不上的方法论描述
- [ ] **信息密度是否合理？** 每 10 行是否产出至少 1 个可操作的检查项或判定标准

判定：
- 核心指令 <30% → 🔴 严重冗余，建议拆分为"检查清单+详细指南"两层
- 无场景裁剪机制 → 🟡 建议增加动态加载触发词
- 教科书式展开 → 🟡 建议提取示例到独立 examples/ 目录

---

## 二、评审过程的 token 纪律

- [ ] **关联上下文按需拉取？** 先列目录 → 按相关性排序 → 只读 top-N（N≤5），而非一次性并行读 10+ 文件
- [ ] **重载阶段隔离？** 预计总 token >30k 时，Phase 3+4 spawn subagent 隔离
- [ ] **报告写文件不写对话？** 完整报告写入文件，主对话只收摘要卡片（≤1k token）
- [ ] **文件读取有明确目的？** 每次 Read 前能回答"这个文件的内容会进入评审结论的哪个部分"
- [ ] **禁止为"全面了解背景"而批量读取？** 评审不需要了解所有背景，只需要验证关键声称

---

## 三、Subagent 隔离判断

逐项检查，3 项全满足 → 启用隔离模式：

- [ ] **预计评审总 token >30k？**（评审对象行数 >200 × 关联文件数 >5 × phase 覆盖 ≥3）
- [ ] **Phase 3/4 包含大量文件读取？**（需要读 5+ 个文件做技术验证或代码审查）
- [ ] **验证操作不需要用户实时介入？**（无"需用户确认后才能继续"的步骤）

**反例：评审对象 <200 行且无 Phase 4（轻量 Skill/文档）→ 禁止隔离。** 隔离模式对此场景会反超（重读税 8-12k + 方法论加载 > 主对话直跑）。此时应走主对话直跑，不要为"演示隔离模式"而手动覆盖路由规则。

### 三-A、Subagent prompt 瘦身模板

隔离模式 subagent prompt 必须用以下精简结构（禁止携带方法论描述）：

```
## Phase 1/2 结论（压缩到 ≤500 字）
- 价值三轴：X/25，核心瓶颈=[一句话]
- 推荐方案：[方案名]，核心理由=[一句话]
- 关键假设：A1[状态] A2[状态]

## 验证清单（纯检查项，无方法论）
- [ ] 维度1：[具体检查项] → 验证文件 path:line
- [ ] 维度2：[具体检查项] → 验证文件 path:line
- [ ] 维度3：[具体检查项] → 验证文件 path:line

## 文件路径
1. path/to/file1.md (行号范围)
2. path/to/file2.md (行号范围)
→ 关联上限 5 个

## 输出
完整报告写入：path/to/report.md
返回摘要卡片（≤1k token）
```

**禁止写入 prompt 的内容**：
- ❌ 角色定义（"你是 XXX 助手"）—— subagent type 系统提示已覆盖
- ❌ 方法论描述（"怎么验证"、"验证三步法"）—— 应由 agent type 预载
- ❌ 格式约束示例（"Spec 原文 + 实测结果 + 差异分析"的模板）—— 输出要求一行即可
- ❌ Phase 1/2 的推导过程（"为什么评这个分"）—— 只传结论

**效果**：prompt 从 ~3000 字 → ~800 字，省 ~40%。

### 三-B、通用隔离纪律

- 必须预传入 Phase 1/2 结论摘要
- 必须传入至少 2 个具体文件路径 + 行号
- 禁止使用"探索"、"查找"、"了解"等开放式指令
- 关联上下文文件限制 ≤5 个
- **禁止在 prompt 中写入方法论或角色定义**（这是"教科书随船携带"的反模式）

---

## 四、摘要卡片要求

主对话只收以下格式的摘要卡片（目标 ≤1k token）：

```markdown
## [评审对象] deep-review 完成
| 阶段 | 评分/结论 | Top 发现 |
|------|----------|---------|
| 价值三轴 | X/25 | [一句话] |
| 方案推荐 | [方案名] | [核心理由] |
| 技术验证 | 🔴N 🟡N 🟢N | [最严重问题] |
| 代码审查 | 🔴N 🟡N 🟢N | [最严重问题] |
→ 完整报告: `path/to/review-report.md`
```

摘要卡片禁止项：
- ❌ 不贴完整报告内容
- ❌ 不逐条列出验证结果
- ❌ 不贴代码片段
- ❌ 不超过 1k token

## 五、命令输出预过滤

Subagent 内执行命令时，必须预过滤噪音输出，防止污染上下文：

### Git 输出过滤

```bash
# 过滤未跟踪文件和 CRLF 警告
git status --short | grep -v "^??" | grep -v "^warning:"

# 安全检查 commit 数后再决定是否执行 diff
git rev-list --count HEAD  # ≥2 才执行 git diff --stat HEAD~1
```

### 通用过滤规则

- [ ] **git status**：排除 `??` 前缀（未跟踪文件），排除 `warning:` 前缀（平台噪音）
- [ ] **git diff**：先检查 commit 数 ≥2，排除 CRLF 警告行
- [ ] **编译输出**：只捕获错误计数和关键错误行，不保留完整编译日志
- [ ] **任何输出 >100 行**：先 grep 关键模式再读取，不全量进上下文
- [ ] **文件 Glob 结果**：>20 个文件时只取前 20 个，标注总数

**反例**：task-snapshot 验证时 `git status --short` 输出 153 行（89 行 `??` 噪音），占 subagent 执行 token 的 80%+。预过滤后应 <30 行。
