# 多视角隔离决策框架

**核心原则**：同一个上下文窗口内的"多视角"本质是同一理解框架下的角色扮演——共享证据基础、受确认偏误影响、产出伪共识。真正的独立视角需要隔离上下文。

---

## 一、隔离必要性决策矩阵

对每次 deep-review，执行以下评分——**在路由阶段（Step 0）强制完成**：

| 信号 | 权重 | 低（1分） | 中（3分） | 高（5分） |
|------|------|----------|----------|----------|
| **决策不可逆性** | ×3 | 随时可改（变量命名、格式调整） | 改有成本但可控（接口设计、模块边界） | 改不了或代价极大（数据库schema、Skill架构、协议定义） |
| **领域交叉度** | ×2 | 单一领域（纯代码、纯设计） | 2个领域交叉（代码+部署、设计+安全） | 3+领域交叉（安全+UX+性能+成本+合规+未来兼容） |
| **确认偏误风险** | ×2 | 用户无倾向，纯探索 | 用户有倾向但不坚持 | 用户已有明确倾向，评审大概率是找认同 |
| **方案空间广度** | ×1 | 只有1条可行路径 | 2-3条已知路径 | 完全开放，可能有未被发现的替代方案 |
| **时间敏感度** | ×1 | 必须几分钟内决策 | 几小时内可以定 | 可以明天再定，不赶时间 |

**计算公式**：`总分 = Σ(信号分 × 权重)`，满分 = 5×3+5×2+5×2+5×1+5×1 = 45

### 阈值与执行模式

| 总分 | 执行模式 | Phase 1+2 在哪跑 | 说明 |
|------|---------|-----------------|------|
| ≤15 | **主对话直跑** | 全部主对话 | 单Agent角色扮演足够，隔离成本>收益 |
| 16-30 | **轻量隔离** | Phase 1 主对话 → Phase 2 用2个对抗subagent | 关键决策需要独立视角，但不需要完全隔离 |
| ≥31 | **全隔离模式** | Phase 1 + Phase 2 都用subagent隔离 | 高不可逆决策，需要完全独立的理解框架 |

### 全隔离模式特殊规则

当总分 ≥31 时，Phase 1 不再在主对话执行。改为：

```
主对话准备 → 整理原始需求/方案摘要 + 关键约束
  → 3个并行 Phase 1 subagent（不同切入点，独立读需求）
     → 每个产出："我认为最关键的3个问题是什么"
  → 主对话：对比3份"关键问题"清单 → 找交集（大概率是真问题）和差集（可能是某个视角的盲区）
  → 确定评审重点 → 进入 Phase 2 多视角隔离
```

## 二、对抗视角选择指南

**关键原则**：选的视角之间必须有天然冲突。三个视角大概率得出相同结论 → 隔离无意义 → 降级为主对话直跑。

| 评审对象类型 | 推荐对抗视角 | 天然冲突点 |
|-------------|-------------|-----------|
| **架构决策**（Skill设计、系统边界、数据流） | ① 极简主义（"最少组件是否可行？"） ② 未来维护者（"6个月后这会崩在哪？"） ③ 替代路径（"有没有完全不同的第三条路？"） | 极简 vs 扩展性天然对立；替代路径挑战前两者的共同假设 |
| **安全/权限设计** | ① 攻击者（"我怎么绕过它？"） ② 合规审计（"满足什么标准？"） ③ 真实用户（"安全措施会不会让我想绕过去？"） | 安全与体验天然冲突；攻击者与合规关注不同层面 |
| **产品/需求决策** | ① 用户价值（"用户真的需要吗？有没有更简单的方案？"） ② 技术成本（"实现这个要多少代价？有没有便宜的方式？"） ③ 业务影响（"不上这个功能会丢客户吗？上了会带来什么新问题？"） | 价值与成本天然对立；短期价值与长期影响可能矛盾 |
| **Skill/Agent 设计** | ① 使用者（"我用这个Skill最大的摩擦是什么？"） ② 维护者（"3个月后这个Skill还能维护吗？有没有隐性复杂度？"） ③ 编排者（"在流水线里它会被怎么调用？接口稳定吗？"） | 使用简单 vs 维护完整 vs 编排标准化 —— 三者不可兼得 |
| **性能优化** | ① 基准测试（"数字上真的快了吗？"） ② 可读性代价（"快了但还能看懂吗？6个月后谁还敢改？"） ③ 边际收益（"这个优化值得做吗？还有没有更值得优化的地方？"） | 性能 vs 可读性天然对立；局部优化 vs 全局最优 |
| **配置/规范设计**（记忆体系、命名约定、目录结构） | ① 新人视角（"从零开始，这个设计最让我困惑的是什么？"） ② 扩展视角（"未来新增类型/项目时，这个设计需要改多少？"） ③ 一致性视角（"和现有规范有没有冲突？所有异常路径都考虑了吗？"） | 简单（新人友好）vs 完整（扩展友好）vs 一致（与历史的兼容） |

## 三、Subagent Prompt 编写规则

**铁律：给每个 subagent 不同的证据基础和推理起点，不只换角色名字。**

### 正确 vs 错误

```
❌ 错误（换帽子不换脑子）:
  Subagent A: "你是架构师。请评审方案A和方案B。"
  Subagent B: "你是安全工程师。请评审方案A和方案B。"
  ↑ 读同样的材料，只是"角色标签"不同
  ↑ 产出相似结论概率高——浪费隔离成本

✅ 正确（不同推理起点）:
  Subagent A: "你是极简主义者。你的原则：能用3个组件解决的不用5个。
    找出方案中可以删除的组件，论证为什么删了也不会影响核心功能。
    默认所有方案都有过度设计。你的使命是砍。"

  Subagent B: "你是未来维护者。你的原则：6个月后接手这个系统的人会恨什么？
    找出方案中会变成技术债的部分。假设今天的实现者已经离职，
    没有文档、没有口头解释。你面对的就是一堆代码/配置。"

  Subagent C: "你是替代方案探索者。你的原则：最好的方案可能还没被列出来。
    你必须提出至少一个方案A和方案B之外的第3方案。
    不需要它完美，只需要它挑战了A和B共同的基本假设。"
```

### Prompt 模板

```markdown
你是 [视角名]（[英文名]）。你的唯一任务是 [单一明确的目标]。

## 你的原则
[1-2句定义你的核心信念——和另一个视角的天然冲突点]

## 你的任务
1. [具体操作1]
2. [具体操作2]
3. 产出：[具体产出格式]

## 输入材料
[关键文件路径 + 行号 —— 不给无关材料]

## 约束
- 你只从你的视角出发做判断，不尝试"全面平衡"
- 你不考虑"但另一个视角会说..."——那是合成阶段的事
- 如果你的视角指向了和材料中不同的方向，大胆指出
```

### 视角数量选择

| 评分 | 视角数 | 理由 |
|------|--------|------|
| 16-22 分 | 2 个 | 轻量隔离，2个对抗视角+主对话合成 |
| 23-30 分 | 2-3 个 | 按领域交叉度决定第3个是否有增量价值 |
| ≥31 分 | 3 个 | 全隔离，Phase 1+2 都用 3 视角 |

## 四、合成规则

### 四步合成流程

```
Step 1: 逐视角提取核心判断
  - 每个 subagent 的推荐结论（一句话）
  - 每个 subagent 的"最关键的1个理由"

Step 2: 交叉验证
  - 多个视角都指出的问题 → 置信度加倍（真问题）
  - 只有一个视角指出 → 标注为该视角特有关注点（可能被遗漏）
  - 多个视角都忽略的维度 → 标注为集体盲区

Step 3: 置信度判定
  - 3/3 视角一致推荐 → 🟢 高置信度
  - 2/3 视角一致，1个分歧 → 🟡 中置信度，必须列出少数意见
  - 1/3 各自推荐不同方案 → 🔴 低置信度，强制要求用户介入
  - 2/2 一致 → 🟡 中置信度（样本小，标注"未经第3视角验证"）

Step 4: 输出合成报告
  - 推荐方案 + 置信度
  - 各视角核心判断（≥1句/视角，标注是共识还是分歧）
  - 分歧点说明（如适用）
  - 集体盲区标注（如适用）
```

### 合成报告模板

```markdown
## Phase 2 合成报告（多视角隔离）

### 推荐结论
- 推荐：[方案X]
- 置信度：🟢高 / 🟡中 / 🔴低
- 理由：[一句话]

### 各视角判断

| 视角 | 推荐 | 最关键的1个理由 | 共识/分歧 |
|------|------|---------------|----------|
| 极简主义 | 方案A | [理由] | 共识 |
| 未来维护者 | 方案A | [理由] | 共识 |
| 替代路径 | 方案C | [理由] | 分歧 |

### 分歧说明（如适用）
[少数意见的具体内容 + 为什么多数否决了它 / 为什么这个问题值得你关注]

### 集体盲区（如适用）
[所有视角都没深入覆盖的维度]
```

## 五、成本模型

| 执行模式 | 预估 token | 适用场景 | 不适用 |
|---------|-----------|---------|--------|
| 主对话直跑 | ~8k（Phase 2） | ≤15分 | ≥16分——省了 token 但漏了独立视角 |
| 轻量隔离 | ~20k（Phase 2） | 16-30分 | ≤15分（过度隔离） |
| 全隔离 | ~35k（Phase 1+2） | ≥31分 | ≤30分（成本过高，收益递减） |

**ROI 判断**：决策不可逆性≥4分 → 即使低分场景也考虑提升一档隔离级别。

## 六、反模式

| 反模式 | 表现 | 修正 |
|--------|------|------|
| **假隔离** | 3个subagent用同一prompt只换角色名 | 不同推理起点 |
| **伪共识** | 合成时忽略少数意见，强行"一致推荐" | 少数意见必须列出 |
| **隔离过剩** | typo修复也启动3视角 | 执行评分，≤15分直跑 |
| **视角固定** | 所有评审都用"架构师+安全+成本" | 按评审对象选视角 |
| **跳过自检** | 隔离模式执行了但没写合成报告 | 合成报告是强制产物 |
