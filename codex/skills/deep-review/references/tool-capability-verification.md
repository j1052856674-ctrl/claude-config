# 工具/Skill 能力验证框架

> 此文件是 `phase3-tech-design.md` 的补充。
> 当评审对象涉及"某工具/Skill 能做什么"的声称时，Phase 3 自动加载。
> **核心原则**：不凭描述相信能力，凭实际运行/源码阅读验证。

---

## 加载条件（自动触发关键词）

当技术文档或评审对象中出现以下任一关键词时，此清单自动生效：

- **Skill 定义**：`Skill`、`SKILL.md`、`provides`、`depends_on`
- **CLI 工具声称**：`npx`、`node.*scripts`、`CLI脚本`、`命令行工具`
- **能力声称**：`确定性`、`deterministic`、`数学级`、`精确计算`
- **工具对比**：`vs`、`替代`、`合并`、`卸载`（当上下文涉及工具/Skill 时）

---

## 维度 0：工具能力验证（5 个子维度）

### 0.1 Prompt-level vs Deterministic Tool 区分

**验证什么**：被评审对象声称的能力是 prompt-level（AI 行为指引）还是 deterministic tool（确定性计算）？两者是否被正确区分？

**怎么验证**：
1. 读取工具的 SKILL.md 或入口文件，列出其能力清单
2. 对每个能力，判定其执行方式：
   - **Prompt-level**：通过文本指令引导 AI 行为，执行质量依赖 AI 理解，无独立可执行逻辑
   - **Deterministic**：有独立可执行脚本/二进制，输入→输出由代码逻辑决定，不依赖 AI 推理
3. 如果一个工具同时有两层能力（如 impeccable：SKILL.md prompt层 + scripts/detector CLI层），必须分别列出

**检查项清单**：

- [ ] 工具的**全部能力清单**已列出（从 SKILL.md / README / 源码提取）
- [ ] 每个能力标注了执行方式：`[P]` prompt-level 或 `[D]` deterministic
- [ ] 如有 CLI 脚本，**实际路径**已确认（不假设 `npx` 包名）
- [ ] 如有 CLI 脚本，**实际调用命令**已确认（如 `node scripts/detect.mjs`，而非猜测的 `npx xxx`）
- [ ] Prompt 层和 CLI 层的**能力边界**已清晰区分

**输出格式**：

| 能力 | 层 | 执行方式 | 调用路径 | 验证状态 |
|------|:--:|---------|---------|:--:|
| 场景驱动代码生成 | Prompt | AI 行为指引 | `/frontend` | ✅ 已读取 |
| WCAG 对比度计算 | CLI | 数学公式 | `node scripts/detector/detect.mjs` | ✅ 已读源码 |
| 设计评审打分 | Prompt | AI 行为指引 | `/impeccable audit` | ✅ 已读取 |

### 0.2 CLI 命令存在性验证

**验证什么**：评审中引用的 CLI 命令是否真实存在？调用路径是否正确？

**怎么验证**：
1. 对评审中出现的每个 CLI 命令声称（如 `npx impeccable audit`），执行存在性检查
2. 如果命令不存在 → 🔴 严重问题，评审结论基于错误前提
3. 如果命令存在但路径不对 → 🟡 中等问题，需修正调用方式

**检查项清单**：

- [ ] 每个 `npx <name>` 声称对应的 npm 包真实存在（或确认不存在）
- [ ] 每个 `node <script>` 声称对应的脚本文件存在且可执行
- [ ] 每个 `/command` 声称对应的 Skill 子命令存在
- [ ] 命令的**实际调用方式**与评审中描述的一致

**常见错误模式**：
- 把 Skill 子命令当成 npm 包（如 `npx impeccable audit` → 实际是 `/impeccable audit`）
- 把内部脚本路径当成公开 CLI（如 `scripts/detect.mjs` 被描述为 `npx tool`）
- 混淆不同层的调用方式（prompt 层和 CLI 层混用同一个命令名）

---

### 0.3 能力覆盖范围实测

**验证什么**：工具声称的能力是否真的覆盖了目标场景？

**怎么验证**：
1. 对 CLI 工具：实际运行，用真实输入测试，捕获输出
2. 对 Prompt 工具：读取其 SKILL.md 和 reference 文件，对比其声称的能力范围与目标需求
3. 对混合工具：分别验证两层

**检查项清单**：

- [ ] CLI 工具已实际运行至少 1 次，输出已捕获
- [ ] Prompt 工具的 reference 文件已读取，能力范围已确认
- [ ] 工具的**实际能力边界**（能做什么、不能做什么）已明确
- [ ] 与评审对象声称的能力覆盖范围对比，差异已标注

---

### 0.4 能力重叠分析

**验证什么**：多个工具/Skill 之间是否存在真实的能力重叠？重叠的程度和性质是什么？

**怎么验证**：
1. 列出每个工具的能力清单（按 0.1 的格式）
2. 逐项对比，标注重叠区域
3. 区分**同层重叠**（两个都是 prompt-level，竞争关系）和**跨层互补**（一个是 prompt，一个是 CLI，协作关系）

**检查项清单**：

- [ ] 每个候选工具的能力清单已列出
- [ ] 重叠区域已标注，并区分：同层竞争 vs 跨层互补
- [ ] 同层重叠 → 需合并/卸载决策
- [ ] 跨层互补 → 定义协作边界，不合并

**关键原则**：
- 两个 prompt-level Skill 覆盖同一能力域 → **冗余**，选一个卸载
- 一个 prompt-level Skill + 一个 CLI 工具覆盖同一目标 → **互补**，定义调用链
- 两个 CLI 工具覆盖同一能力域 → 对比精度/覆盖度后选一个

---

### 0.5 工具集成可行性验证

**验证什么**：如果评审结论是"工具 A 编排工具 B"，这个集成在技术上是否可行？

**怎么验证**：
1. 确认 B 的调用接口（API / CLI / Skill 子命令）
2. 确认 A 能否实际调用 B（路径、参数、输出解析）
3. 测试端到端调用链（如可行）

**检查项清单**：

- [ ] B 的调用接口已确认（CLI 路径 + 参数格式 或 Skill 子命令名）
- [ ] A 调用 B 的方式已明确（depends_on 声明 / subagent 调用 / shell 命令）
- [ ] B 的输出格式与 A 的输入期望匹配
- [ ] 调用链无循环依赖

## 常见失误模式

### 模式 T1：Prompt 层和 CLI 层混淆

**表现**：用 CLI 层的确定性能力为 Prompt 层的子命令辩护，或反之。

**例子**：评审说 "impeccable audit 能做数学级对比度测量"——实际上 `audit` 是 prompt 子命令，对比度计算在 `detect.mjs` CLI 中。两者是独立的能力。

**验证方法**：读取工具的 SKILL.md 子命令定义 + 读取 scripts 源码，确认每个能力属于哪一层。

**识别信号**：评审中用一个工具名指代两个不同的能力层（"X 工具能做 Y"但实际上 Y 是 X 的 CLI 脚本做的，不是 X 的 prompt 子命令做的）。

---

### 模式 T2：npx 包名假设

**表现**：假设工具的 CLI 入口是 `npx <skill-name>`，但实际工具没有发布为 npm 包。

**例子**：`npx impeccable audit` → 实际 impeccable 是 Claude Code Skill，CLI 脚本通过 `node .codex/skills/impeccable/scripts/detect.mjs` 调用。

**验证方法**：检查工具的 package.json 是否有 `bin` 字段，或直接找 scripts 目录下的可执行入口。

**识别信号**：评审中出现 `npx <名称>` 但该名称不在 npm registry 中。

---

### 模式 T3：能力声称未实际运行验证

**表现**：接受工具文档中的能力描述为事实，没有实际运行或读取源码。

**例子**：评审说 "这个工具能做 X"，但评审者没有运行工具也没有读源码。

**验证方法**：至少执行以下之一——① 运行 CLI 工具并捕获输出；② 读取核心源码确认算法逻辑；③ 读取 SKILL.md + reference 确认 prompt 层能力范围。

**识别信号**：评审报告中有关于工具能力的肯定性陈述，但无"验证操作"栏或验证操作写的是"根据文档描述"而非"实际运行/读源码"。

---

### 模式 T4：跨层互补被误判为同层重叠

**表现**：两个工具一个做 prompt-level 生成、一个做 CLI 确定性验证，评审把它们当成同一层的竞争关系。

**例子**：frontend v2（prompt 层生成）和 impeccable CLI（确定性检测）被放在"该留哪个"的二选一框架中，但它们其实不是竞争关系。

**验证方法**：先执行 0.1 区分，再执行 0.4 重叠分析。

**识别信号**：评审的对比表中，两个不同层级的工具被放在同一行比较。
