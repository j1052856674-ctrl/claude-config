# Phase 3: 技术落地验证

> 目标：让技术方案从"纸面正确"变成"运行正确"。
> 本文件不回答"验证什么结论"，而是回答"怎么验证"。具体结论来自实测。

---

## 深度模式说明

Phase 3 支持三种运行深度，由文档内容自动决定：

| 模式 | 触发条件 | 验证范围 |
|------|---------|---------|
| **纯文档路径** | 文档中无 CLI 命令 / Skill能力 / 工具声称 等关键词 | 通用维度（数据模型完整性、接口/数据结构、数据流与状态、错误处理） |
| **CLI 集成路径** | 文档中出现 CLI 命令 / spawn / exec / API 调用 / session / resume / 认证 等关键词 | 通用维度 + 5维度 CLI 验证框架（参数组合、输出解析、进程生命周期、会话上下文、平台兼容） |
| **工具能力路径** | 文档中出现 Skill / provides / depends_on / npx / 确定性 / CLI脚本 / 工具能力 等关键词 | 通用维度 + 工具能力验证框架（prompt-vs-deterministic、CLI存在性、能力覆盖实测、重叠分析、集成可行性） |

动态深度触发规则：
- CLI 路径触发正则：`\b(claude\s+-|spawn\s*\(|exec\s*\(|execSync|child_process)` / `\b(api\s*key|token|auth|credential|oauth|jwt\b)` / `\b(session[_-]?id|--resume|--session|persistent.session)` / `\b(platform|windows|linux|macos|posix|process\.exit|SIGTERM|SIGKILL)`
- 工具能力路径触发正则：`\b(Skill|SKILL\.md|provides|depends_on|npx|确定性|deterministic|CLI脚本|工具能力|卸载|合并决策)\b`

两类关键词同时出现 → 两条路径都启用。详见 `references/tool-capability-verification.md`。

## 零、上下文感知验证判定（Phase 3 启动时必执行）

**目标**：避免重复验证对话中已有的调研结论，同时不跳过真正需要验证的推测性声明。

### 判定流程

```
Q1: 评审对象是否在同一对话中，基于本对话已读取/执行过的内容产出？
  → 否（外部文档/跨会话文档）：
    → 执行标准验证流程（下面"一"开始的全流程），无豁免
  → 是：
    → 进入 Q2

Q2: 评审对象是"总结型"还是"设定型"？
  - 总结型：内容是对对话中已读取文件/已执行命令的归纳重组，没有超出已有证据的新声称
    例子：基于已读取的26个文件写出的标准化方案、基于已执行测试输出的覆盖率报告
  - 设定型：内容包含对话中未验证的技术假设、设计推断、行为猜测
    例子："这个 CLI 参数应该支持组合"、"API 应该返回 JSON"、"这样设计应该不会冲突"

→ 总结型：
  - 验证证据来源 = 对话历史中已有的读取/执行结果
  - 标注"基于对话内已有调研，跳过文件重读"
  - 但：方案中任何"未验证的假设/推测/行为声称"仍按设定型处理
→ 设定型：
  - 执行标准验证流程
  - 重点验证对话中未覆盖的推测性声明
```

### 判定示例

| 场景 | 判定 | 动作 |
|---|---|---|
| 对话中已读取10个记忆文件→写出的记忆管理标准化方案 | 总结型 | 引用对话调研结果，跳过重读。但方案中"200行上限合适"这类主观断言需讨论验证 |
| 基于对话中 pattern 观察→推断出"CLI 参数设计模式"的技术 Spec | 设定型（包含行为推断） | 执行标准验证，重点验证 CLI 参数推断是否正确 |
| 打开对话前已存在的 PRD/设计文档 | 外部文档 | 执行标准验证流程，无豁免 |
| 对话中跑完测试→写出的测试覆盖率总结 | 总结型 | 引用对话执行结果，跳过重跑 |

### 关键原则

- **对话历史中有证据 = 可以直接引用**：不必为了"独立验证"重新读取相同文件
- **推测性声称必须验证**：即使是同一对话产出的方案，只要是"应该/可能/预计"类推断，就须验证
- **判定不确定时默认定型**：宁可多做验证，不因误判跳过必要检查

## 一、验证方法论核心原则

### 1. 验证优先级

| 证据来源 | 可信度 | 适用场景 |
|----------|--------|----------|
| 实际运行验证 | 最高 | CLI 参数、输出格式、进程行为、平台兼容 |
| 官方文档（最新版本） | 中高 | API 行为、已知限制（但文档也可能滞后） |
| 网络搜索/社区讨论 | 中 | 版本兼容性、已知 bug、最佳实践 |
| 个人记忆/推测 | 低 | 仅作为验证线索，**不能作为评审结论** |

### 2. 验证三步法

对于 Spec 中每个技术声称：

```
1. 提取声称 → "Spec §6.1 说命令是 claude -p "task" --output-format=stream-json --bare"
2. 实际运行 → 执行该命令，捕获完整 stdout/stderr
3. 对比差异 → 输出是否与 Spec §6.2 的解析逻辑匹配？
```

如果实际运行失败（报错、输出格式不对、行为与声称不同），这就是 🔴 严重问题。

## 二、5维度 CLI 验证框架

> 当动态深度触发 CLI 路径时，按以下 5 个维度逐条验证。
> 每个维度保留"验证什么 + 怎么验证 + 检查项清单"。

---

### 维度 1：参数组合正确性

**验证什么**：Spec 中写出的 CLI/API 命令拼接模板能否实际运行？参数组合是否有隐含依赖？

**怎么验证**：
1. 复制 Spec 中的每个命令模板，用真实值替换变量占位符
2. 实际执行每个命令，记录 stdout 前 50 行 + stderr
3. 逐个去掉参数，测试哪些去掉后报错（隐含依赖）
4. 对比工具 `--help` 输出，找出 Spec 漏掉的必需参数
5. 测试边界值（空值、超长值、特殊字符）

**检查项清单**：

- [ ] Spec 中每个命令模板都能实际运行（不报参数错误）
- [ ] 参数间的隐含依赖已标注（某些 flag 必须搭配另一个 flag）
- [ ] Spec 漏掉的必需参数已发现
- [ ] 参数值边界行为已测试

**执行步骤**：
1. 读取 Spec 中所有 CLI 命令模板
2. 拼接实际可执行的命令（用真实值替换变量占位符）
3. 逐个运行，记录每个命令的 stdout（前 50 行）和 stderr
4. 逐个去掉参数，测试哪些是隐含必需的
5. 对比 `--help` 输出与 Spec 使用的参数列表

---

### 维度 2：输出解析逻辑与实际输出一致性

**验证什么**：Spec 假设的输出格式与实际输出是否一致？解析逻辑能否处理所有实际出现的事件类型？

**怎么验证**：
1. 实际运行简单任务 + 输出格式参数，捕获 100% 输出
2. 实际运行需要 tool use 的任务，捕获 100% 输出
3. 实际运行会失败的任务，捕获 100% 输出
4. 提取所有实际出现的 type/subtype 清单
5. 对比 Spec 的解析逻辑覆盖范围
6. 检查 message.content 内部嵌套结构

**检查项清单**：

- [ ] Spec 描述的输出事件类型全覆盖（实际出现的每个 type 都有处理逻辑）
- [ ] message.content 内部子类型已区分（text / tool_use / thinking / tool_result）
- [ ] 内部循环事件（tool use 循环产生的 user↔assistant 交替）有正确处理
- [ ] 初始化事件（session_id、工具列表等）有提取和存储逻辑
- [ ] 进度/状态增量事件有处理或过滤策略
- [ ] JSON 解析失败降级策略已定义
- [ ] 最终 result 事件的 success/error 有正确映射到 Worker 状态

**执行步骤**：
1. 运行简单任务（纯文本输出），捕获 100% stream-json
2. 运行需要 tool use 的任务，捕获 100% stream-json
3. 运行会失败的任务，捕获 100% stream-json
4. 分析每种场景的完整 type/subtype 清单
5. 对比 Spec 的解析逻辑是否能处理所有发现的类型
6. 标记 Spec 未覆盖的事件类型为 🔴

---

### 维度 3：进程生命周期管理

**验证什么**：Spec 的进程管理逻辑能否覆盖实际的 spawn→运行→退出→kill 全链路？

**怎么验证**：
1. spawn 正常任务 → 等退出 → 记录 exit code
2. spawn 必定失败任务 → 等退出 → 记录 exit code
3. spawn 任务 → 中途 kill → 检查残留进程（ps/tasklist）
4. 在目标平台上测试 kill 方式（kill -TERM / taskkill /F / taskkill /T /F）
5. 测试 stdin 管道关闭对进程的影响

**检查项清单**：

- [ ] exit code = 0 映射 completed，非 0 映射 failed
- [ ] kill 时进程树清理覆盖目标平台（Windows: /T flag；Unix: kill 子进程）
- [ ] stdin 管道在 kill 前已关闭
- [ ] stuck 检测基于"无新输出时间"而非纯超时
- [ ] kill 后有等待确认步骤（3秒 wait）再清理数据
- [ ] 进程异常崩溃（无 result 事件）有外层监控兜底

**执行步骤**：
1. spawn 正常任务 → 等退出 → 记录 exit code + 输出
2. spawn 必定失败任务 → 等退出 → 记录 exit code + 输出
3. spawn 任务 → 中途 kill → 检查残留进程
4. spawn 任务 → 观察长时间无输出 → 测试 stuck 检测
5. 对比 Spec 的状态映射与实测结果

---

### 维度 4：会话与上下文管理

**验证什么**：session_id 的提取→存储→使用链路是否完整？上下文恢复机制是否可行？

**怎么验证**：
1. 运行任务 → 在输出中定位 session_id（哪个事件、哪个字段）
2. 提取 session_id → 执行 --resume → 验证上下文恢复
3. 检查数据模型是否有 session_id 字段
4. 检查 ~/.claude/sessions/ 目录行为（是否默认保存、是否需要手动清理）
5. 测试 compaction 触发条件（极长任务）

**检查项清单**：

- [ ] session_id 可从输出提取，数据模型有存储字段
- [ ] --resume 可恢复上下文，Spec 有使用场景描述
- [ ] Compaction 机制已说明（工具内部处理 vs 集成方需自行处理）
- [ ] session 持久化行为已说明（默认保存 vs --no-session-persistence）
- [ ] kill Worker 时是否需要清理 session 文件有明确策略
- [ ] 多 Worker 并行时 session_id 管理不冲突

**执行步骤**：
1. 运行任务 → 从 stream-json 提取 session_id
2. 用 session_id 执行 --resume → 验证上下文恢复
3. 用 --session-id 指定自定义 ID → 验证可控
4. 检查 Spec 是否描述了 session_id 的提取、存储、使用全链路

---

### 维度 5：平台兼容性

**验证什么**：在目标平台上实际测试路径、编码、进程清理、信号处理。

**怎么验证**：
1. 在目标平台运行 where/which 找到工具安装路径
2. 测试 Spec 的路径探测逻辑能否找到实际安装位置
3. 在目标平台 spawn 进程 + kill，验证进程清理方式
4. 测试中文路径、特殊字符路径的兼容性
5. 测试编码（UTF-8 输出在目标平台是否正常）

**检查项清单**：

- [ ] CLI 安装路径探测覆盖目标平台的实际位置
- [ ] 进程清理方式覆盖目标平台差异（Windows: taskkill /T /F）
- [ ] 文件路径使用 PathBuf 或等价类型处理
- [ ] 中文路径/特殊字符兼容性已测试
- [ ] stdout 编码在目标平台正确处理

## 三、通用维度（所有技术文档适用）

无论是否触发 CLI 路径，以下通用维度都必须验证。

---

### 维度 6：数据模型完整性

**验证什么**：后端和前端数据模型是否覆盖了所有有用的输出数据？两端是否对齐？

**怎么验证**：
1. 对比实际 CLI result 事件的字段列表与 Spec WorkerInfo 字段
2. 对比 Rust enum/struct 与 TypeScript type/interface 的每个字段
3. 检查枚举值是否完全对齐（包括 serde rename 与 TS literal）

**检查项清单**：

- [ ] 状态/枚举用真正的 enum + serde rename，不是裸 String
- [ ] Rust 和 TS 的每个字段名、类型、枚举值完全对齐
- [ ] CLI result 事件中的用量数据（token、cost、duration）有对应字段
- [ ] LogEntry 的 level 分类覆盖实际输出的所有事件类型
- [ ] 环形缓冲区（VecDeque）而非 Vec 用于输出缓冲（如果存储最近 N 行）

---

### 维度 7：接口 / 数据结构

**验证什么**：输入输出定义是否明确？命名是否一致？是否向后兼容？

**怎么验证**：
1. 检查每个接口的输入参数类型、必填/可选、默认值
2. 检查返回结构体的每个字段是否有明确类型和语义
3. 检查变更是否破坏已有接口契约

**检查项清单**：

- [ ] 输入输出明确（无 any/unknown 逃逸）
- [ ] 命名一致（同一概念在不同层使用相同名称）
- [ ] 向后兼容（新增字段有默认值，删除字段有迁移说明）
- [ ] 枚举值有完整清单，无魔法字符串

---

### 维度 8：数据流与状态

**验证什么**：数据流向是否单向清晰？状态所有权是否明确？多源数据是否有冲突处理？

**怎么验证**：
1. 画数据流图，标注每个节点的输入/输出/副作用
2. 检查状态修改是否集中管理
3. 检查并发场景下数据一致性

**检查项清单**：

- [ ] 数据流单向清晰（无双向绑定导致的循环更新）
- [ ] 谁拥有谁修改（Ownership 明确）
- [ ] 多源冲突有处理策略（最新优先 / 合并 / 报错）
- [ ] 状态变更可追溯（有日志或事件记录）

---

### 维度 9：错误处理

**验证什么**：正常路径、空路径、极端路径、权限路径是否都有处理？

**怎么验证**：
1. 对每个接口/函数，按 4 类路径设计测试用例
2. 检查错误处理是否覆盖所有已知异常场景
3. 检查错误信息是否对用户友好

**4 类路径检查**：

| 路径 | 检查点 | 标记 |
|------|--------|------|
| 正常路径 | 标准输入 → 期望输出 | ✅ |
| 空路径 | 空输入 / 空数组 / 空字符串 | ✅ |
| 极端路径 | 超长 / 超大 / 超快 / 超慢 | ✅ |
| 权限路径 | 未授权 / 越权 / 过期 Token | ✅ |

**错误处理矩阵模板**：

| 场景 | 输入 | 期望行为 | 实际行为 | 状态 |
|------|------|---------|---------|------|
| 正常输入 | ... | ... | ... | ✅ |
| 空输入 | ... | ... | ... | ✅ |
| 超长输入 | ... | ... | ... | ✅ |
| 权限不足 | ... | ... | ... | ✅ |

**检查项清单**：

- [ ] 正常路径已验证
- [ ] 空路径已验证（空输入、空数组、空字符串）
- [ ] 极端路径已验证（超长、超大、超快、超慢）
- [ ] 权限路径已验证（未授权、越权、过期 Token）
- [ ] 错误信息对用户友好（非内部堆栈暴露）
- [ ] 有降级策略（主路径失败时的 fallback）

### 触发 CLI 路径的关键词清单

| 类别 | 关键词示例 |
|------|-----------|
| CLI 命令 | `claude -p`, `claude --output-format`, `--bare`, `--resume` |
| 进程操作 | `spawn`, `exec`, `execSync`, `child_process`, `fork` |
| API 调用 | `fetch`, `axios`, `apiKey`, `token`, `OAuth`, `JWT` |

### 判定流程

```
1. 读取文档全文
2. 扫描是否包含上述关键词
3. 若包含 → 启用 CLI 路径（通用维度 + 5维度CLI验证框架）
4. 若不包含 → 走纯文档路径（仅通用维度）
```

## 五、验证纪律

1. **禁止凭记忆验证**：所有结论必须有运行截图或日志支撑。个人记忆/推测可信度最低，仅作为验证线索。
2. **版本标注**：所有验证结果必须标注工具版本、验证日期、验证环境。后续使用者可判断结论是否仍有效。
   - **工具版本**：如 `Claude Code v2.1.168`
   - **验证日期**：如 `2026-06-08`
   - **验证环境**：如 `Windows 10, npm global install`
3. **mistake-patterns 提醒**：评审中若发现与已知错误模式匹配的问题，引用 `mistake-patterns.md` 中的对应条目，并在评审报告中标注引用编号。

```markdown
**实测结果**：得到了 `具体输出/行为`

**与 Spec 声称的差异**：[描述差异，或写"无差异"]

**严重度**：🔴 / 🟡 / 🟢

**修正建议**：[具体改法，或写"无需修正"]
```
