# 深度配置与裁剪路由

---

## 场景类型→裁剪路线完整表格

| 场景 | Phase 1 | Phase 2 | Phase 3 | Phase 4 |
|---|---|---|---|---|
| 小需求(<1天) | ✅完整 | ✅简化版(3维度) | ❌跳过 | ❌跳过 |
| 中等需求(1-3天) | ✅完整 | ✅完整 | ❌跳过 | ✅(有代码时) |
| 大需求(>3天) | ✅完整 | ✅完整 | ✅完整 | ✅完整 |
| 纯代码改动 | ✅最小版(确认范围) | ❌跳过 | ❌跳过 | ✅完整 |
| 紧急修复 | ✅最小版(根因+影响) | ❌跳过 | ❌跳过 | ✅简化版(逻辑+安全+YAGNI) |
| 技术Spec评审 | ✅完整 | ✅完整 | ✅完整+CLI清单 | ❌跳过 |
| PRD评审 | ✅完整 | ✅完整 | ❌跳过 | ❌跳过 |

---

## 场景冲突优先级

当多个场景标签同时命中时，按优先级取裁剪路线：

**紧急修复 > 纯代码改动 > 中等需求 > 小需求**

即：紧急修复场景始终用紧急修复的裁剪路线，不再叠加其他路线。

## 执行模式路由表

根据场景和预估 token 消耗，决定重载阶段在主对话跑还是隔离进 subagent：

| 场景 | 执行模式 | Phase 3+4 执行位置 | 预估主对话 token |
|------|---------|-----------------|------------------|
| 小需求(<1天) | 主对话直跑 | Phase 3+4 已裁剪 | <10k |
| 紧急修复 | 主对话直跑 | Phase 3 已裁剪，Phase 4 简化版主对话 | <12k |
| PRD 评审 | 主对话直跑 | Phase 3+4 已跳过 | <8k |
| 中等需求(无代码) | 主对话直跑 | Phase 3+4 已跳过 | <12k |
| 中等需求(有代码) | **混合模式** | Phase 4 进 subagent | <15k（主对话） |
| 大需求(>3天) | **隔离模式** | Phase 3+4 合并进 subagent | <15k（主对话） |
| 技术Spec评审 | **隔离模式** | Phase 3+4 合并进 subagent | <15k（主对话） |
| 纯代码改动 | 主对话直跑 | Phase 3 已跳过，Phase 4 主对话 | <15k |

### 执行模式判定规则

```
IF 场景 in {大需求, 技术Spec}:
  → 隔离模式（总 token 预估 >30k）
ELIF 场景 == 中等需求 AND 有代码:
  → 混合模式（Phase 4 隔离）
ELSE:
  → 主对话直跑（总 token 预估 <15k）
```

### 隔离模式 subagent prompt 结构（精简版）

```
## Phase 1/2 结论（≤500字）
- 价值三轴：X/25，核心瓶颈=[一句话]
- 推荐方案：[方案名]

## 验证清单（纯检查项）
- [ ] 维度1：[检查项] → path:line
- [ ] 维度2：[检查项] → path:line

## 文件（≤5个，全部用绝对路径）
1. E:/project/path/to/file1.md (L1-L50)

## 输出
写报告到绝对路径: E:/project/08_Reports/skill-deep-review/target/review-report.md
返回摘要卡片（≤1k token）
```

> 详细瘦身模板见 `context-efficiency.md` 三-A。
> **路径纪律**：文件和输出路径必须传绝对路径，禁止依赖 CWD——subagent 的 CWD 可能与主对话不同。

### 强制直跑规则（禁止手动覆盖）

以下场景**强制主对话直跑，禁止手动启用隔离模式**：

| 条件 | 理由 | 反例 |
|------|------|------|
| 评审对象 <200 行且无 Phase 4 | 隔离模式"重读税"8-12k + 方法论加载 → 总 token 反超主对话直跑 | task-snapshot(110行)走隔离 → 57k vs 预估直跑 15k |
| Phase 3+4 均已裁剪 | 无重载阶段，隔离无收益 | 小需求(Phase 3+4 跳过)仍走隔离 |
| 关联文件 ≤3 个 | 重读税占比 >50%，隔离亏本 | 仅评审单个 SKILL.md + 原型 |

**判定口诀**：评审对象 <200 行 + 关联文件 ≤5 + 无 Phase 4 → 不隔离。

### 摘要卡片格式（主对话收）

```markdown
## [评审对象] deep-review 完成
| 阶段 | 评分/结论 | Top 发现 |
|------|----------|---------|
| 价值三轴 | X/25 | [一句话] |
| 方案推荐 | [方案名] | [核心理由] |
| 技术验证 | 🔴N 🟡N 🟢N | [最严重问题] |
| 代码审查 | 🔴N 🟡N 🟢N | [最严重问题] |
→ 完整报告: `path/to/review-report.md`
```
（目标：≤1k token）

## Phase 4 简化版定义

紧急修复时 Phase 4 只检查 3 个维度：

| 维度 | 检查什么 |
|---|---|
| 逻辑正确性 | 边界条件、空值、并发、状态转换 |
| 安全 | 敏感信息、注入/XSS、权限校验 |
| YAGNI | 是否引入未使用的功能/过度抽象 |

跳过维度：性能、测试充分性、VC断言覆盖、代码规范。

## Phase 3 动态深度触发规则

### 自动检测关键词

文档中出现以下关键词时，自动启用 CLI/API 集成评审路径（加载 `cli-verification.md`）：

| 类别 | 关键词示例 |
|---|---|
| CLI命令拼接 | `claude -p`, `--output-format`, `--verbose`, `--permission-mode`, `--bare`, `--resume` |
| 进程管理 | `spawn`, `exec`, `child_process`, `Command::new`, `process.spawn`, `subprocess` |
| 外部API调用 | `fetch`, `HttpClient`, `request`, `axios`, `curl`, `API endpoint` |
| session/resume/认证 | `--session-id`, `--resume`, `auth`, `token`, `keychain`, `OAuth`, `API Key` |

### 路径选择

- **检测到关键词** → 全维度（参数组合+输出解析+进程+会话+平台）+ 加载 `cli-verification.md`
- **未检测到关键词** → 通用维度（数据模型+接口+数据流+错误处理）

## 场景化权重调整模板

| 评审场景 | 权重调整 | 调整理由 |
|---|---|---|
| PRD评审 | UX 30%, 可行性 15% | PRD核心是用户价值 |
| 技术 Spec评审 | 可行性 30%, 扩展性 20%, UX 10% | 技术文档关注能否做出来 |
| 底层架构评审 | 一致性 25%, 性能 20%, 扩展性 20% | 架构核心是稳定+性能 |
| 0→1新项目 | 扩展性 25%, 可行性 25% | 最怕做不出来和后期重构 |
| 成熟项目增量 | 性能 20%, 一致性 20%, UX 25% | 关注体验+不退化 |

默认权重（7维度）：UX 25% + 可行性 20% + 扩展性 15% + 性能 10% + 学习成本 10% + 错误恢复 10% + 一致性 10%
